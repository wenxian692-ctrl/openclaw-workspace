## 1. 文档目的

本文档定义好享健康 APP 中"监督与调整"模块的完整产品方案，供产品、前端、后端、AI、数据、测试、顾问团队评审与落地。

本模块不是单纯的提醒系统，而是 AI Agent 驱动健康管理的控制层，负责在全生命周期内持续感知用户状态，识别信号，生成动作，并验证动作是否生效。

**本模块覆盖阶段**：
- 首次测评
- 报告后继续获取方案
- 方案生成与待激活
- 付费待支付、待发货、待签收
- Day0 与 Day1
- 28 天服务执行期
- Day7/14/21 周复评
- Day28 阶段复评
- 下阶段承接
- 服务终止与退款评估

## 2. 模块定义

监督与调整模块的核心职责是：

1. **持续监督** - 贯穿用户全生命周期
2. **信号识别** - 从事件中提取有效信号
3. **分数更新** - 维护三大核心分数
4. **生命周期切换** - 智能判断用户所处阶段
5. **调整策略生成** - 基于信号和分数生成动作
6. **触达与执行** - 选择合适渠道触达用户
7. **结果回写** - 记录并验证调整效果
8. **用户反馈收集** - 任何时点用户主动反馈的收集和处理

**统一主链路**：

```
监督来源 → 事件 → 信号 → 分数 → 生命周期 → 干预等级 → 调整动作 → 触达 → 执行 → 回写 → 再观察
```

## 3. 模块目标

### 3.1 用户目标
- 用户始终知道当前最重要的下一步
- 用户在困难时能被及时理解和拉回
- 用户不会因平台期、低能量、场景波动被误判
- 用户在高风险时能被及时保护和升级人工
- 用户的任何主动反馈都能被及时响应和处理

### 3.2 业务目标
- 提升测评完成率
- 提升方案激活率
- 提升 Day0 与 Day1 完成率

- 提升周复评与阶段复评完成率
- 降低掉队率、投诉率、无反馈率

### 3.3 系统目标
- 可解释
- 可追溯
- 可配置
- 可灰度
- 可审计

## 4. 核心设计原则

### 4.1 全生命周期持续监督
监督不是只发生在执行期，而是贯穿以下所有阶段：
- 测评前和测评中
- 方案待获取与待激活
- 履约等待期
- Day0、Day1
- 执行期
- 复评期
- 阶段切换期
- 终止和退款评估期

### 4.2 显性与隐性并行
监督必须同时覆盖两类信息：
- **显性监督**：用户明确完成、明确反馈、明确表达
- **隐性监督**：不打卡、不回复、临近截止才完成、打开不执行、短回复、重复同一困难

### 4.3 用户主动反馈优先
- 任何阶段用户主动表达的负向情绪、方案调整需求、任务调整需求
- 都是最高优先级的监督输入
- 必须立即响应，不能等待定时检测

### 4.4 规则、AI、混合判定分层
- **规则引擎**：负责明确阈值、明确时点、必须稳定的判断
- **AI**：负责自然语言理解、阻碍识别、语义归因
- **混合判定**：负责 AI 结构化提取后由规则引擎落动作

### 4.5 风险优先级最高
任何阶段都遵循以下优先级：
1. 医疗和安全边界
2. 用户主动反馈（负向情绪、调整需求）
3. 正在进行中的测评和复评
4. 当前主任务闭环
5. 风险纠偏
6. 日常主动问询
7. 商业转化和续费

### 4.6 调整必须回写和验证
系统不能只"调一次"，必须记录：
- 为什么调
- 调了什么
- 用户是否接受
- 调整后 3 天或 7 天是否改善
- 是否继续、恢复、升级或终止

## 5. 决策思维链

监督与调整模块的决策必须按以下9步执行：

### Step 1：识别当前所处阶段
先判断用户当前在哪个阶段：
- 首次测评 → 报告后待继续 → 方案待激活 → 付费待支付 → 付费待签收 → Day0 → 执行期 → 周复评 → 阶段复评 → 下阶段承接 → 终止

**注意**：系统不能跨阶段误触发。

### Step 2：收集监督事件
从显性事件、隐性事件、用户主动反馈中收集当前有效监督输入。

### Step 3：生成信号
对事件做归并和解释，得到策略可用的信号。

### Step 4：更新分数
更新：执行分、能量分、风险分

### Step 5：判断生命周期
根据分数、趋势、目标达成情况更新：S1/S2/S3/S4/S5

### Step 6：判定干预等级
根据信号和分数判定：L0/L1/L2/L3

### Step 7：选择动作和触达渠道
根据干预等级和具体场景选择动作类型和触达渠道。

### Step 8：执行与触达
通过首页卡片/站内消息/Push/短信/人工触达用户。

### Step 9：回写与效果验证
验证窗口建议按动作类型分层：
- 轻提醒：24 小时
- 任务调整：3 天
- 行为任务调整：3 天或 1 个周期
- 周复评后微调：7 天
- 人工干预：由人工 SLA 驱动

## 6. 三大分数体系

| 分数名称 | 默认值 | 核心用途 | 驱动内容 |
|----------|--------|----------|----------|
| **执行分** (execution_score) | 60 | 判断是否在执行 | 任务升级、降级、缩包；S1/S2/S4切换 |
| **能量分** (energy_score) | 70 | 判断做不做得动 | 运动和行为任务轻重；AI照护语气 |
| **风险分** (risk_score) | 0 | 判断掉队、流失、投诉、安全风险 | L1/L2/L3；人工介入决策 |

## 7. 生命周期定义

### 7.1 S1 新手启动期

| 项目 | 内容 |
|------|------|
| 进入条件 | 注册后到方案激活后的前7天，或关键数据尚未完整 |
| 重点 | 补档、Day0/Day1引导、降低摩擦 |

### 7.2 S2 稳定执行期

| 项目 | 内容 |
|------|------|
| 进入条件 | 执行分>=65，风险分<30，无平台期和高危风险 |
| 重点 | 稳定执行、适度升级、强化反馈 |

### 7.3 S3 平台期风险

| 项目 | 内容 |
|------|------|
| 进入条件 | 真实平台期成立，执行分>=60 |
| 重点 | 平台期解释、饮食方法微调、防焦虑和流失 |

### 7.4 S4 风险期

| 项目 | 内容 |
|------|------|
| 进入条件 | 风险分>=50 或 执行分<45 或 连续3天无核心任务完成 或 信任危机/高危表达 |
| 重点 | 拉回、缩包、Plan B、必要时人工 |

### 7.5 S5 稳态维持期

| 项目 | 内容 |
|------|------|
| 进入条件 | 达到阶段目标，稳定维持7到14天 |
| 重点 | 频率下调、防反弹、维持习惯 |

## 8. 平台期判定规则

### 8.1 真实平台期（True Plateau）

**判定条件**（需同时满足）：

| 条件 | 阈值 | 说明 |
|------|------|------|
| 连续体重数据 | ≥7天 | 每日有体重记录 |
| 体重波动 | <0.2kg | 7天平均体重与前一天差值 |
| 执行分 | ≥60 | 任务执行正常 |
| 能量分 | ≥60 | 用户能量状态正常 |
| 无明显阻碍主诉 | - | 用户未表达明显困难 |

**处理策略**：
- 不扣执行分
- 触发平台期解释（AI）
- 饮食方法微调候选
- 防焦虑流失

### 8.2 假平台期（False Plateau）

**判定条件**（满足任一）：

| 条件 | 阈值 | 说明 |
|------|------|------|
| 连续体重缺失 | ≥3天 | 无体重数据 |
| 执行分 | <60 | 任务执行不达标 |
| 能量分 | <50 | 用户能量状态差 |
| 隐性掉队信号 | ≥2次 | 不打卡、不回复等 |
| 阻碍主诉 | 明确表达 | 用户明确表达困难 |

**处理策略**：
- 按正常风险处理
- 执行分/能量分正常扣减
- 优先解决执行问题
- 必要时L2干预

## 11. 分数规则详解

### 11.1 执行分（execution_score）

**默认值**：60

#### 11.1.1 加分项

| 加分项 | 分值 | 触发条件 |
|--------|------|----------|
| 当日主任务按时完成 | +2/项 | 当日21:00前完成（每日最多+6） |
| 连续3天完成今日三件事 | +5 | 连续3天完成率100% |
| 连续7天完成率>=80% | +8 | 连续7天达标 |
| 主动补录有效数据 | +1 | 补录过往数据且有效（每日最多+2） |
| 有效回复主动问询 | +1 | AI判定为有效回复 |
| 完成当周复评 | +8 | 周复评完成 |
| 隐性加分：真平台期保护 | 0 | 处于S3真平台期时不扣分 |

#### 11.1.2 减分项

| 减分项 | 分值 | 触发条件 |
|--------|------|----------|
| 单项任务逾期 | -2/项 | 超过截止时间完成 |
| 单项任务未完成 | -4/项 | 当日未完成 |
| 连续2天核心任务未完成 | -6 | 连续2天 |
| 连续3天核心任务未完成 | -12 | 连续3天（触发S4） |
| 追问已触发但无回复 | -2 | 24h内无回复 |
| 低质量打卡 | -2 | AI判定质量低 |
| 周复评触发后48h未开始 | -6 | 超时未开始 |
| 隐性减分：假平台期正常扣 | - | 处于S3假平台期按正常规则 |

#### 11.1.3 使用原则
- 真平台期不扣执行分
- 用户主动说明客观困难时，扣减减半
- 投诉和医疗边界不走执行分路径

## 12. 干预等级体系

| 等级 | 名称 | 适用场景 | 动作类型 | 响应时限 |
|------|------|----------|----------|----------|
| **L0** | 不干预 | 风险低、执行稳定 | 无 | - |
| **L1** | 提醒 | 单次未完成、单次数据缺失、轻度拖延 | 首页卡片、站内消息、AI轻提醒、Push | <30min |
| **L2** | 调整 | 连续偏差、低能量、平台期、多次提醒无效 | 缩小任务包、任务降级、替代任务、行为替换、饮食微调、周末模式、Plan B | <2h |
| **L3** | 人工 | 高危红线、医疗边界、投诉退费、连续L2无效 | 顾问接管、工单、电话/企微、医学或营养师介入 | **即时** |

**触发规则**：
- risk_score >= 80 → 直接 L3
- 用户主动反馈负向情绪/求助 → 即时L1/L2/L3
- 高危红线和医疗边界不等待二次确认

## 13. 用户主动反馈收集与处理

> **核心原则**：用户在任何阶段、任何流程、任何闲聊时主动表达的负向情绪、方案调整需求、任务调整需求，都是最高优先级的监督输入，必须立即响应。

### 13.1 用户主动反馈类型

| 反馈类型 | 信号Code | 触发场景 | 响应优先级 |
|----------|----------|----------|------------|
| 负向情绪表达 | user_negative_emotion | 用户说"太难了"、"想放弃"、"心情不好"、"压力大"等 | **P0-即时** |
| 方案调整请求 | user_plan_adjust_request | 用户说"想换方案"、"目标完不成"、"想调整计划"等 | **P0-即时** |
| 任务调整请求 | user_task_adjust_request | 用户说"任务太多了"、"这个做不来"、"想换个任务"等 | **P1-<1h** |
| 用户求助 | user_help_request | 用户说"怎么办"、"帮帮我"、"有問題"等 | **P0-即时** |
| 正向反馈 | user_positive_feedback | 用户说"很有用"、"效果很好"、"继续保持"等 | 记录存档 |

### 13.2 用户主动反馈处理流程

```
用户主动输入
    ↓
AI实时语义识别（判断是否为负向/调整/求助）
    ↓
是 → 生成对应信号 → 立即触发干预
    ↓
否 → 判断为闲聊/正向反馈 → 正常响应或存档
```

### 13.3 不同反馈类型的处理策略

#### 13.3.1 负向情绪处理

| 场景 | 处理策略 |
|------|----------|
| 轻度负向（偶尔抱怨） | AI共情+鼓励，次日轻关注 |
| 中度负向（持续沮丧） | AI共情+任务减负（L2），48h内持续关注 |
| 重度负向（想放弃/自暴自弃） | AI共情+立即缩小任务包+人工关注准备 |
| 高危表达（涉及安全） | 不等待，直接升级L3人工 |

#### 13.3.2 方案调整请求处理

| 场景 | 处理策略 |
|------|----------|
| 用户觉得目标太高 | AI解释当前进度合理性，若坚持，收集归因后进入方案调整流程 |
| 用户觉得方案不适合 | AI追问具体原因，判断是执行问题还是方案问题 |
| 用户想换方案 | 记录原因，评估是否满足换方案条件，进入方案调整流程 |

#### 13.3.3 任务调整请求处理

| 场景 | 处理策略 |
|------|----------|
| 任务太多做不完 | 立即缩小任务包（从3件减到2件） |
| 任务太难 | 立即难度调整或行为替换 |
| 任务不想做 | 追问原因，根据原因选择调整或替换 |
| 时间安排不合理 | 调整提醒时间和任务窗口 |

#### 13.3.4 用户求助处理

| 场景 | 处理策略 |
|------|----------|
| 一般问题 | AI即时回答，记录为知识库候选 |
| 复杂问题 | AI回答后标记，48h后回访确认 |
| 医疗/专业问题 | 立即升级L3人工 |

### 13.4 用户反馈的分数影响

| 反馈类型 | 对执行分 | 对能量分 | 对风险分 |
|----------|----------|----------|----------|
| 负向情绪 | - | -5~15 | +5~20 |
| 方案调整请求 | 记录 | - | +10 |
| 任务调整请求 | 记录 | - | - |
| 用户求助 | - | - | - |

### 14.3 行为打卡监督

#### 14.3.1 监督指标（四个评分维度）

| 维度 | 权重 | 说明 | 分值范围 |
|------|------|------|----------|
| 真实性 | 30% | 是否真实执行 | 1-5分 |
| 执行质量 | 25% | 动作是否规范 | 1-5分 |
| 效果感知 | 20% | 是否感受到效果 | 1-5分 |
| 可持续性 | 25% | 能否长期坚持 | 1-5分 |

**行为评分公式**：
```
behavior_score = 真实性×0.30 + 执行质量×0.25 + 效果感知×0.20 + 可持续性×0.25
```

#### 14.3.2 应用维度分数映射

| behavior_score范围 | 判定 | 建议动作 | 优先级 |
|--------------------|------|----------|--------|
| >=4.2 | 优秀 | 继续巩固/升级难度 | 低 |
| 3.4~4.2 | 良好 | 保持当前 | 中 |
| 2.6~3.4 | 一般 | 难度调整 | 中 |
| 1.8~2.6 | 较差 | 行为替换 | 高 |
| <1.8 | 极差 | 任务移除 | 最高 |

#### 14.3.3 3天周期评估矩阵

| 完成情况 | 判定 | 处理动作 |
|----------|------|----------|
| 0/3或1/3完成，无回复 | 隐性掉队 | 退出执行位→降P3→次日切换下一任务 |
| 0/3或1/3完成，用户明确拒绝 | 核心动作合理 | 难度调整 |
| | 核心动作不合理 | 行为替换 |
| | 用户明确拒绝方向 | 任务移除 |
| 2/3完成但Day3失败 | - | 难度调整→无效再替换 |
| 3天完成，score>=4.2 | 优秀 | 流转下一任务，正向标签弱化 |
| 3天完成，3.4<=score<4.2 | 良好 | 保持，不升级 |
| 3天完成，2.6<=score<3.4 | 一般 | 难度调整 |
| 3天完成，1.8<=score<2.6 | 较差 | 行为替换（新任务进P1） |
| 3天完成，score<1.8 | 极差 | 任务移除 |

### 14.4 任务调整动作类型定义

#### 14.4.1 难度调整

**定义**：在不改变目标和核心动作的前提下调整参数。

**可调整参数**：
- 次数（frequency）
- 频次（interval）
- 时长（duration）
- 执行窗口（time_window）
- 执行场景（context）

**示例**：
| 原任务 | 调整后 |
|--------|--------|
| 饭后站立10分钟 | 饭后站立3分钟 |
| 每天3次深呼吸 | 每天1次 |
| 睡前1小时关手机 | 提前15分钟 |

#### 14.4.2 行为替换

**定义**：AI基于原任务目标生成更好执行的新任务。

**约束**：
- 目标不变
- 成本更低
- 更适配当前场景

**替换后**：
- 新任务进入P1顶部
- 原任务标记为replaced

#### 14.4.3 任务移除

**条件**（满足任一）：
- Day1明确反馈不合适
- 连续>=3天不打卡且不回应
- score<1.8
- 用户明确拒绝

**必须执行**：
- 解释原因（AI）
- 用户确认

**移除后**：
- 次日切换下一任务
- 标记为removed

## 16. 行为干预任务详细策略

### 16.1 任务定位

| 任务类型 | 执行周期 | 目标 |
|----------|----------|------|
| 数据追踪类 | 1天 | 获取数据，结束后即停止 |
| 行为习惯类 | 3天 | 形成习惯，Day3后输出评分和动作 |

### 16.2 Day1/Day2/Day3目标

| 阶段 | 目标 |
|------|------|
| Day1 | 确认是否真实执行；判断是否明显不适配 |
| Day2 | 判断执行质量；判断是否开始产生效果；识别具体阻碍 |
| Day3 | 判断是否可形成习惯；形成最终行为评分 |

### 16.3 评分维度详解

| 维度 | 问题示例 | 打分标准 |
|------|----------|----------|
| 真实性 | "你今天真的做了吗？" | 5=完全真实，1=完全虚假 |
| 执行质量 | "动作做得标准吗？" | 5=非常标准，1=完全不对 |
| 效果感知 | "你感觉到效果了吗？" | 5=效果明显，1=没感觉 |
| 可持续性 | "你能长期坚持吗？" | 5=完全能，1=完全不能 |

### 16.4 行为评分与动作映射

| 评分 | 等级 | 动作 | 执行策略 |
|------|------|------|----------|
| >=4.2 | 优秀A | 继续/升级 | 保持或提升难度 |
| 3.4~4.2 | 良好B | 保持 | 不调整 |
| 2.6~3.4 | 一般C | 难度调整 | 降低参数 |
| 1.8~2.6 | 较差D | 行为替换 | 生成新任务 |
| <1.8 | 极差E | 任务移除 | 移除并切换 |

### 16.5 行为任务优先级

| 优先级 | 说明 | 调度规则 |
|--------|------|----------|
| P1 | 最高 | 新任务、替换任务、用户指定任务 |
| P2 | 中 | 常规习惯任务 |
| P3 | 低 | 降级任务、补卡任务 |

| 调整类型 | 适用范围 | 调整幅度 |
|----------|----------|----------|
| 目标调整 | 体重目标、周期目标 | ±5-10% |
| 方案微调 | 饮食方案、运动方案 | 参数级调整 |
| 方案切换 | 主方案更换 | 需复评后决策 |
| Plan B | 备用方案激活 | 立即执行 |

### 17.3 方案调整流程

```
用户/系统触发方案调整
    ↓
收集原因（执行问题/方案问题/客观因素）
    ↓
判断是否满足调整条件
    ↓
是 → 生成调整建议 → 用户确认 → 执行
    ↓
否 → 解释原因 → 维持当前方案
```

### 17.4 周期内调整限制

| 调整类型 | 周期内次数限制 | 生效时间 |
|----------|----------------|----------|
| 目标调整 | 1次/28天 | 即时生效 |
| 方案微调 | 2次/28天 | 次日生效 |
| 方案切换 | 1次/阶段 | 复评后生效 |
| Plan B激活 | 不限 | 即时生效 |

## 19. 全生命周期分阶段 PRD

### 19.1 阶段A：首次测评

| 项目 | 内容 |
|------|------|
| 覆盖状态 | NOT_STARTED, IN_PROGRESS, INTERRUPTED, COMPLETED, EXPIRED, CANCELED |
| 监督重点 | 是否进入/中断/完成测评，是否出现非服务适配风险 |
| 显性监督 | assessment.start/interrupt/complete |
| 隐性监督 | 多次进入又退出(>=3次)、长时间停留不提交、连续极短回复 |
| **核心信号(中文)** | 测评抵抗(assessment_resistance)、测评超时(assessment_timeout)、不可服务风险(service_ineligible_risk) |
| **用户反馈处理** | 任何测评期间负向表达→即时响应+适当引导 |
| 核心动作 | 首次中断→首页恢复卡+AI轻召回；48h内第二次→简化测评建议；72h未继续→降低催促；高风险→终止+解释 |

### 19.2 阶段B：报告已出，待继续获取方案

| 项目 | 内容 |
|------|------|
| 监督重点 | 是否理解报告、是否愿意继续、是否被成本/难度/信任问题阻断 |
| 显性监督 | 报告打开、继续获取方案点击 |
| 隐性监督 | 多次看报告但不继续、高频问"有用吗" |
| **核心信号(中文)** | 方案犹豫(plan_hesitation)、信任危机(trust_crisis)、求快心理(urgency_bias) |
| **用户反馈处理** | 任何方案相关负向/调整需求→即时AI解释+引导 |
| 核心动作 | 优先解释报告→AI解答"为何需要继续"→必要时首页卡承接 |

### 19.3 阶段C：方案前阶段测评P1

| 项目 | 内容 |
|------|------|
| 监督重点 | 是否完成补充信息、是否中断、是否疲劳 |
| 核心动作 | 中断后恢复入口；AI追问补齐；多次中断→减少问题量 |

### 19.4 阶段D：方案已出未激活/付费未支付

| 项目 | 内容 |
|------|------|
| 覆盖状态 | READY, AWAITING_PAYMENT |
| 监督重点 | 是否确认方案、是否激活免费方案、是否发起支付但未完成 |
| 显性监督 | 查看方案、方案确认、点击支付 |
| 隐性监督 | 多次看方案不激活、发起支付后24h不完成 |
| **核心信号(中文)** | 方案犹豫(plan_hesitation)、支付放弃(payment_abandon) |
| **用户反馈处理** | 支付犹豫→即时解释权益；不想付→问原因 |
| 核心动作 | 免费→解释后促激活；付费→解释不夸大；支付未完成→站内卡+Push |

### 19.5 阶段E：付费后待发货/待签收

| 项目 | 内容 |
|------|------|
| 覆盖状态 | AWAITING_DELIVERY |
| 监督重点 | 发货/物流/签收是否正常 |
| 核心规则 | 实时回调为主，对账轮询兜底，签收后次日进入Day1 |
| 核心动作 | 发货成功→即时Push；签收成功→即时Push+Day1预告 |

### 19.6 阶段F：激活后待Day0

| 项目 | 内容 |
|------|------|
| 监督重点 | 是否完成Day0准备、是否具备执行条件 |
| 显性监督 | Day0页面打开、Day0任务完成 |
| 隐性监督 | 多次打开Day0但不执行 |
| **核心信号(中文)** | Day0卡住(day0_stuck) |
| **用户反馈处理** | Day0期间任何困难→即时响应+调整 |
| 核心动作 | Day0只强调准备；截止前轻提醒；未做完→次日首页承接 |

### 19.7 阶段G：执行期Day1-Day28

| 项目 | 内容 |
|------|------|
| 监督重点 | 今日三件事是否执行；是否出现掉队/低能量/场景等 |
| 核心规则 | 正常→L0/轻反馈；单次未完成→L1；连续未完成→L2；高风险→L3 |
| **用户反馈处理** | **全周期最高优先级**：任何负向/调整/求助→即时响应 |
| 打卡模块监督 | 详见第14节 |
| 日常问询监督 | 详见第15节 |

### 19.8 阶段H：Day7/14/21周复评

| 项目 | 内容 |
|------|------|
| 监督重点 | 是否点击/完成复评；目标进度是否达标 |
| 核心规则 | 08:00出口；20:00未开始→Push；+24h未开始→再次提醒；+48h未完成→超时 |
| **复评后可调整** | 饮食方法微调、任务参数微调、行为优先级调整、运动强度轻调 |
| **复评后不可做** | 服务阶段切换、下一阶段方案生成 |

### 19.9 阶段I：Day28阶段复评

| 项目 | 内容 |
|------|------|
| 监督重点 | 是否进入复评；是否达标；未达标原因 |
| 核心规则 | Day28 08:00出口；20:00未开始→Push；+24h→Push+首页强提醒；+48h→L2 |
| **达标处理** | 当前阶段完成→进入下一阶段→生成下一阶段方案 |
| **未达标处理** | 不直接进入下一阶段→先归因→判断服务是否充分交付→服务终局判定→必要时退款 |
| **归因类型** | A1-执行不足、A2-饮食方法不匹配、A3-能量/生理波动、A4-场景干扰、A5-产品/履约问题 |

### 19.10 阶段J：终止与退款评估

| 项目 | 内容 |
|------|------|
| 覆盖状态 | TERMINATED |
| 终止原因 | Day28未达标归因终局、提前终止规则命中、高风险/投诉 |
| 终止后监督 | 退款进度、投诉闭环、人工跟进、用户是否点击重新开始 |
| 终止后停止 | 不再生成任务、不再发Push、不再推进阶段升级 |

### 20.1 标签状态

| 状态 | 说明 |
|------|------|
| suspected | 怀疑/待确认 |
| confirmed | 已确认 |
| intervening | 干预中 |
| weakened | 弱化中 |

### 20.2 标签流转规则

| 流转 | 条件 |
|------|------|
| 进入 | 测评命中/主动承认/AI明确识别/隐性信号>=2次 |
| 升级 | 7天内同类重复/任务失败/隐性和显性同时命中 |
| 弱化 | 1周期score>=4.2且7天无负向信号 |
| 解决 | 2周期表现良好 或 14天无信号 且 数据不恶化 |
| 保留 | 任务未完成/评分低/用户明确拒绝/隐性信号持续 |
