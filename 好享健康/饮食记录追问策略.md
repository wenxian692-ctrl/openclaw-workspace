# 饮食记录追问策略

### 1. 需求概述

当用户通过拍照或文字方式提交饮食记录，但AI识别结果不清晰（图片部分识别/识别失败、文字模糊描述）时，系统需启动多轮追问流程，引导用户补充信息，直至达到可记录标准或满足终止条件。

本文档详细定义追问流程的各环节话术、处理逻辑及终止条件，作为《AI识别热量》主文档的配套说明。

### 2. 追问流程总览

追问流程分为四个阶段，按顺序执行：

1.  **图片质量问题优先排除**（仅图片输入）
    
2.  **第一步：确认食物清单**
    
3.  **第二步：逐项量化**
    
4.  **第三步：补全隐形热量**
    

特殊场景下，可根据识别到的内容直接跳转至 **7.特殊场景追问清单**

### 3. 图片质量问题优先排除

当用户提交图片且识别置信度不足时，AI首先判断是否为图片质量问题，并采用对应话术引导用户重新拍摄或补充。

| 识别困境 | AI追问话术 | 底部按钮 | 说明 |
| --- | --- | --- | --- |
| **食物被遮挡/角度不佳** | “请调整角度，让餐盘完整入镜，或补充一张俯拍图” | **【重新拍照】** | 需重新拍照改善画质，同时允许用户用文字描述 |
| **光线过暗/滤镜失真** | “检测到光线不足，请关闭美食滤镜后重拍，或开启闪光灯” | **【重新拍照】** | 需重新拍照改善画质，同时允许用户用文字描述 |
| **混合菜肴难以拆分** | 移至“第一步：确认食物清单”处理（见4.2），无需重新拍照 | **无** | 无需重拍，通过文字描述拆分 |

**交互说明**：

*   用户点击【重新拍照】，直接进入食物拍照页面，详阅【[拍照记录](http://axure.sys.hxsapp.net/KHN5C5/#id=bassr8&p=%E6%8B%8D%E7%85%A7%E8%AE%B0%E5%BD%95&g=1)】说明
    
*   重新提交图片后，AI重新进入识别流程，新图片覆盖原图片的识别上下文
    
*   若用户选择不重新拍照，可继续通过文字描述方式记录
    

### 4. 第一步：确认食物清单

**追问必须基于已有信息，而不是从头开始。**

*   **图片输入**：AI已识别出部分食物，追问要围绕“未识别部分”或“模糊部分”
    
*   **文字输入**：用户已提供初步描述，追问要围绕“缺失的关键信息”
    
*   **图文结合**：需比对一致性，围绕矛盾或遗漏部分追问
    

#### 4.1 基于输入源的差异化初始话术

| 输入类型 | 识别情况 | 初始话术原则 | 示例 |
| --- | --- | --- | --- |
| **图片** | 部分识别 | 先呈现已识别食物，再询问遗漏 | “我看到图片里有南瓜饼和鸡蛋，还有其他食物吗？” |
| **图片** | 识别失败 | 说明未识别成功，引导描述 | “图片不太清晰，能告诉我这餐吃了哪些食物吗？” |
| **文字** | 模糊描述 | 基于用户已说内容，追问缺失信息 | “你提到吃了炒菜和主食，具体是什么炒菜？主食吃了什么？” |
| **文字** | 遗漏关键信息 | 针对性追问 | “你说了吃牛肉，主食吃了什么呢？” |
| **图文结合** | 描述与图片不符 | 指出矛盾，引导确认 | “图片看起来有火锅，但你描述的是苹果，能再确认一下吗？” |

#### 4.2 用户回复场景处理

| 用户回复场景 | AI处理 |
| --- | --- |
| **补充清晰完整**（如“米饭、红烧肉、炒青菜、番茄蛋汤”） | 更新食物清单，直接进入第二步量化 |
| **仍模糊**（如“就是普通炒菜”） | 针对性追问：“是什么菜？有肉吗？什么肉？” |
| **明显遗漏**（照片中有汤/饮料/水果但未提及） | “看到旁边有碗汤/杯子/水果，那是什么？喝了多少？” |
| **混合菜肴难以拆分**（如“炒面/炒饭/烩菜”） | “里面能吃到哪些食材？有肉丁、虾仁、鸡蛋、青菜吗？” |
| **零食/加餐混淆**（时间非正餐点） | “这是正餐还是加餐？之前2小时内还吃过别的吗？” |
| **文字描述与图片矛盾**（AI已有图片信息，但用户文字描述与图片明显不符） | “照片看起来比较丰富，和您说的不太一样，能再确认一下吗？” |
| **表示记不清** | 根据现有信息进入估算流程 |

### 5. 第二步：逐项量化

#### 5.1 功能概述

进入量化阶段后，AI根据用户已确认的食物清单，采用**优先级驱动**策略进行追问。不是所有食物都需要追问，也不是所有维度都需要细化——只追问对热量估算最关键的食物和最核心的信息维度，确保在有限的交互中最大化信息价值。

#### 5.2 追问优先级计算

AI根据以下两个维度，为每项食物计算追问优先级：

| 维度 | 说明 | 高优先级条件 |
| --- | --- | --- |
| **热量密度** | 单位重量热量高低 | 主食、肉类、油脂类 > 蔬菜、汤饮 |
| **信息缺失度** | 当前信息是否足够估算 | 无重量/数量/大小信息 > 已有部分信息 |

**优先级规则**：

*   **最高优先级**：热量密度高且完全无量（如肉类无量）
    
*   **次高优先级**：热量密度高但已有部分信息（如“一碗米饭”但不知碗大小）
    
*   **较低优先级**：热量密度低且完全无量（如蔬菜无量）
    
*   **最低优先级**：热量密度低且信息基本够用
    

#### 5.3 追问顺序策略

在追问过程中，AI按以下策略确定每轮追问的食物：

| 追问顺序 | 分配策略 |
| --- | --- |
| **优先追问** | 从剩余未追问的食物中，选择优先级最高的1-2项进行追问 |
| **持续追问** | 每完成一轮，重新计算剩余食物的优先级，继续追问当前最高优先级的食物 |
| **动态调整** | 若用户在某轮中主动提供了额外信息（如一次性说了多道菜的量化），AI需重新评估食物清单，更新待追问列表 |

**设计意图**：确保信息价值最大化——无论追问持续几轮，永远是先问最重要、最缺失的信息。追问何时终止由第8章“追问终止条件”统一判断，本阶段只负责“按什么顺序问”。

#### 5.4 核心追问维度

每类食物只保留最影响热量估算的1-2个核心维度，避免过度追问：

| 食物类型 | 核心维度1 | 核心维度2 | 示例话术 |
| --- | --- | --- | --- |
| 主食 | 份量（碗/个） | 大小参照 | “米饭是家里常用碗还是外卖圆盒？平平一碗还是堆起来的？” |
| 肉类 | 数量（块/只） | 肥瘦比例 | “红烧肉大概几块？是麻将块大小还是拇指大小？肥的多还是瘦的多？” |
| 炒菜 | 份量（碟/盘） | 油量感知 | “炒青菜是浅浅一小碟还是满满一大盘？油多吗？” |
| 汤饮 | 容器大小 | 糖度/配料 | “汤是一小碗还是一大碗？奶茶几分糖？有珍珠吗？” |
| 蔬菜 | 份量（碟/盘） | —— | “沙拉大概有多少？是一小碗还是一大盆？” |

#### 5.5 智能跳过规则

| 场景 | 处理方式 |
| --- | --- |
| 用户已提供明确克数（如“100g牛肉”） | 跳过该食物所有追问 |
| 用户描述包含明确参照（如“一碗米饭”） | 跳过或仅追问容器规格 |
| 热量密度低的食物（如蔬菜）且信息基本够用 | 跳过或合并到最后一轮简单确认 |
| 某类食物已追问过且用户已回复 | 不再重复追问同类食物 |

### 6. 第三步：补全隐形热量

#### 6.1 功能说明

在完成主要食物的量化追问后，AI需评估当前信息是否足够生成精确记录：

*   **若信息已充分**（所有主要食物均有明确量化或清晰参照），则直接进入记录生成环节，不再进行第三步追问。
    
*   **若仍存在影响热量估算的隐形变量**（如烹饪方式、用油量、分食比例、隐形碳水等），则选择性追问最相关的1-3个因素，让估算更精准。
    

本阶段旨在填补剩余的信息缺口，而非无差别追加提问。

#### 6.2 追问原则

*   **按需选择**：根据用户已提供的食物类型和对话上下文，选择性追问最相关的1-3个隐形因素，不必全部提问。
    
*   **自然融入**：引入话术不固定，可根据前序对话灵活变化，避免机械感。
    
*   **尊重用户**：若用户表现出不耐烦或已回答过相关内容，及时跳过。
    

#### 6.3 隐形因素追问列表

| 因素类型 | 适用场景 | AI追问示例 |
| --- | --- | --- |
| **烹饪方式** | 当用户提到炒菜、肉类、油炸类食物时 | “这菜是清炒、红烧、油炸还是凉拌？看起来油亮吗？” |
| **调味轻重** | 当用户提到红烧、糖醋、勾芡类菜肴，或含酱料的菜品时 | “口味偏清淡还是偏咸？有勾芡、淋酱或撒糖吗？” |
| **进食比例** | 当用户明显是多人聚餐场景（如照片显示多个餐具）或菜品量大时 | “这盘菜是您一个人吃完的，还是多人分食？您吃了大概几分之几？” |
| **隐形碳水** | 当用户未提及饮料/水果，但照片中可见，或餐次可能包含此类时 | “除了正餐，有喝含糖饮料、果汁，或吃糕点、水果吗？” |

### 7. 特殊场景追问清单

当AI识别到以下特殊场景时，可替代或补充上述通用追问流程。

| 场景 | AI追问 |
| --- | --- |
| **火锅/烧烤/自助餐** | “锅底是清汤还是红油？吃了哪些肉？有吃主食吗？蘸料是油碟还是干碟？” |
| **聚餐/宴席** | “大概吃了几筷子每道菜？有喝酒吗？主食是最后吃的吗？” |
| **便利店/快餐** | “具体买了什么？有看包装上的热量表吗？饭团/三明治吃了几个？” |
| **代餐/减脂餐** | “是买的成品还是自制？蛋白质含量感觉够吗？几点吃的？” |
| **情绪性进食** | “这餐是因为饿了吃，还是因为压力大/无聊/想吃点好的？” |
| **深夜进食** | “是晚餐没吃饱，还是习惯性夜宵？吃完多久睡觉？” |

### 8. 追问终止条件

#### 8.1 功能说明

本追问流程**不设硬性的轮次上限**，而是以“信息是否足够”为核心判断标准，结合用户交互状态动态终止。原因如下：

*   **用户差异**：有的用户一次说清所有信息（0轮），有的用户需要逐项引导（可能5-6轮）。
    
*   **食物复杂度差异**：单一食物与满桌菜肴所需追问轮次自然不同。
    
*   **体验优先**：硬性截断可能导致记录不准，损失信任；智能判断更符合拟人助手定位。
    

**实际控制手段**：

*   通过“信息足够”“边际效用低”等条件主动收尾，避免无限追问。
    
*   通过“连续无效回复”“用户不耐烦”等条件及时止损。
    
*   通过追问优先级（5.2）确保关键信息优先获取。
    

#### 8.2 终止条件列表

在以下任一情况下，追问流程终止，AI根据已有信息生成记录：

| 终止条件 | 判断标准 | AI处理话术 |
| --- | --- | --- |
| **用户主动终止** | 用户表示“记不清了/差不多就行” | 停止追问，根据现有信息进行估算，并告知用户：“饮食记录可能存在偏差，不利于体重管理，建议尽量回忆完整哦～” |
| **信息足够** | 已追问的食物覆盖了预估总热量的85%以上<br>（**注**：85%为建议阈值，实际可算法动态调整。） | 主动结束：“这些信息已经够啦，我们先记录。如果有遗漏可以随时补充。” |
| **边际效用低** | 剩余未追问的食物热量密度低，追问收益小 | 主动结束，不再追问次要食物，直接生成记录。 |
| **所有食物追问完成** | 用户已提供所有必要信息，无需继续追问 | 生成记录，结束追问。 |
| **连续无效回复** | 用户连续2次追问后仍不配合（如重复回复无效信息、答非所问） | 停止追问，根据现有信息进行估算，并告知用户：“饮食记录可能存在偏差，不利于体重管理，后续可以补充。” |
| **用户表现出不耐烦** | 用户回复简短、消极（如“随便”“就这样吧”） | 根据已有信息估算，并提示偏差风险。 |
| **超时** | 单次追问交互超过5分钟 | 主动结束：“先记录这些，稍后营养师再和您确认。” |

**说明**：以上终止条件共同作用，确保在信息充分时及时收尾，在用户不配合时及时止损，避免无限追问。

### 9. 通用规则

#### 9.1 话术语气

*   保持亲切、鼓励的语气，避免机械感
    
*   适当使用表情符号增强亲和力（如🍚、🥩、🥗）
    

#### 9.2 上下文维护

*   每次追问前，AI需携带已确认的信息，避免重复提问
    
*   用户主动提供额外信息时，AI应动态调整追问方向
    
*   若用户主动返回修改之前的回答，AI应更新对应食物的信息并重新计算优先级。
    

#### 9.3 变量替换

*   话术中的食物名称、数量描述等需根据用户实际情况动态替换
    
*   示例：若用户已说“吃了红烧肉”，追问时可直接说“这红烧肉大概几块？”
