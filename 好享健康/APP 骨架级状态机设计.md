# APP 骨架级状态机设计

## 《好享健康》APP 骨架级状态机设计

### 一、设计总则

*   **主脉络**：以 **服务方案对象** 为核心主状态机，代表用户在一个28天周期内的服务进程。其他对象（测评、任务、干预、支付、日常问询）作为辅助，通过 **事件总线** 与主脉络协同。
    
*   **多对象独立**：每个业务对象拥有独立状态机，职责单一，状态定义遵循 **MECE** 原则。
    
*   **事件驱动**：所有状态变化均通过事件触发，系统解耦，易于扩展。
    
*   **用户状态分离**：用户对象包含 **服务状态**（active/terminated）和 **生命周期状态**（S1~S5），后者仅在服务中有效。
    
*   **主聊天通道调度**：各类系统对话有明确优先级，通过顶部状态栏和调度器协调，确保核心流程不被阻塞。

### 二、核心业务对象状态机定义

#### 2.1 主状态机：服务方案对象（每个28天周期一个实例）

| 状态 | 名称 | 描述 |
| --- | --- | --- |
| **PENDING** | 待生成 | 周期开始，等待阶段测评完成后生成具体方案内容（起始态） |
| **READY** | 已生成待激活 | 方案内容已生成，等待用户点击“开始执行”或发起支付 |
| **AWAITING\_PAYMENT** | 待支付 | 付费方案已生成，用户已发起支付，等待支付结果 |
| **AWAITING\_DELIVERY** | 待发货/待签收 | 付费方案支付成功，等待实物产品发货和用户签收 |
| **ACTIVE** | 激活中 | 方案已激活，每日任务据此生成 |
| **ADJUSTED** | 已调整 | 在周期内被周复评微调（内容变化，仍ACTIVE） |
| **SUPERSEDED** | 已被替换 | 同周期内被升级方案替换（旧方案进入此终态） |
| **COMPLETED** | 已完成 | 周期结束（28天到达），等待下一周期决策 |
| **TERMINATED** | 终止 | 周期未达标或因故提前终止 |

**属性**：

*   `plan_id`：方案ID、
    
*   `user_id`：用户ID、
    
*   `period`：服务方案周期（P1~P4）、
    
*   `plan_type`：服务方案类型：free/paid）
    
*   `generated_by_assessment_id`：生成此方案的阶段测评ID、
    
*   `activated_at`：方案激活时间戳=Day0、
    
*   `execution_start_date`：方案开始执行日期=Day1、
    
*   `last_adjustment_type`：最近调整类型，如_intensify/fine\_tune_、
    
*   `status`：服务方案的状态
    

#### 2.2 测评流程对象（每次测评独立实例）

| 状态 | 名称 | 描述 |
| --- | --- | --- |
| **NOT\_STARTED** | 未开始 | 测评已触发，等待用户进入（起始态） |
| **IN\_PROGRESS** | 进行中 | 用户正在答题 |
| **INTERRUPTED** | 中断 | 用户中途退出，进度已保存 |
| **COMPLETED** | 已完成 | 提交生成报告 |
| **EXPIRED** | 过期 | 超时未完成（48小时） |
| **CANCELED** | 取消 | 取消测评（强迫终止） |

**属性**：

*   `assessment_id`：测评ID、
    
*   `user_id`：用户ID、
    
*   `type`：`first`（初评）、`weekly`（周复评）、`phase`（阶段测评，用于生成新周期的服务方案）、
    
*   `phase`：`type=phase`有效，记录服务方案周期（P1/P2/P3/P4）、
    
*   `is_phase_end`：`type=weekly`有效，标识是否为第28天周期结束复评、
    
*   `report_id`：测评报告ID
    
*   `status`：测评的状态
    

#### 2.3 每日任务对象（每个任务独立实例）

| 状态 | 名称 | 描述 |
| --- | --- | --- |
| **PENDING** | 待执行 | 任务已生成，等待用户执行（起始态） |
| **COMPLETED** | 已完成 | 用户按时完成 |
| **OVERDUE** | 超时 | 超过截止时间未完成 |
| **SKIPPED** | 跳过 | 用户主动跳过（如申请休息日） |
| **ADJUSTED** | 已调整 | 任务内容被调整，仍需执行 |

**属性**：

*   `task_id`：任务ID、
    
*   `plan_id`：计划ID、
    
*   `task_type`：饮食打卡/运动打卡/行为干预、
    
*   `scheduled_date`：任务日期、
    
*   `due_time`：最晚完成时间（如当天23:59），用户设置超时判定、
    
*   `content`：打卡内容、
    
*   `inquiry_status`：追问状态，
    
*   `feedback_status`：反馈状态，`pending`（待追问）、`in_progress`（追问中）、`wait_feedback`（待反馈）、`completed`（已完成）、`aborted`（放弃）
    
*   `feedback_attempts`：已尝试追问次数（默认0）
    
*   `feedback_expire_time`：追问截止时间（首次追问后+1h）
    
*   `feedback_data`：JSON，存储收集到追问与反馈的数据
    
*   `last_asked_at`：最后回复日期
    
*   `status`：任务的状态
    

**备注：**如有子任务`sub_task.plan_id`（每子任务独立），可以参考主任务结构，其属性中含主任务`task_id`作为关联用。

#### 2.4 干预流程对象（每次干预独立实例）

| 状态 | 名称 | 描述 |
| --- | --- | --- |
| **DETECTED** | 风险识别 | 监督模块触发，干预实例创建（起始态） |
| **DECIDING** | 决策中 | AI决策干预方式 |
| **REMINDING** | 提醒中 | L1提醒（Push、短信等） |
| **ADJUSTING** | 调整中 | L2策略调整（Plan B） |
| **HUMAN** | 人工介入 | L3人工 |
| **RESOLVED** | 已解决 | 干预结束 |
| **ESCALATED** | 已升级 | 升级干预等级 |

**属性**：

*   `intervention_id`：干预任务ID、
    
*   `user_id`：用户ID、
    
*   `trigger_event`：触发事件（如`task.overdue`）、
    
*   `risk_level`：风险等级（如L1/L2/L3）、
    
*   `status`：干预任务状态、
    
*   `created_at`：创建时间、
    
*   `resolved_at`：解决时间
    

#### 2.5 支付订单对象

| 状态 | 名称 | 描述 |
| --- | --- | --- |
| **CREATED** | 已创建 | 订单生成（起始态） |
| **PAID** | 已支付 | 支付成功 |
| **FAILED** | 支付失败 | 支付失败 |
| **REFUNDED** | 已退款 | 退款成功 |
| **EXPIRED** | 已过期 | 超时未支付（30分钟） |

**属性**：

*   `order_id`：订单ID、
    
*   `user_id`：用户ID、
    
*   `plan_id`：服务方案ID、
    
*   `amount`：支付金额、
    
*   `status`：支付状态
    

#### 2.6 用户对象

| 字段 | 取值 | 描述 |
| --- | --- | --- |
| `user_id` | \- | 唯一标识 |
| `service_status` | `active` / `terminated` | 是否正在服务周期中 |
| `lifecycle_state` | `S1`~`S5` | 仅在`active`时有效 |
| `pending_questions` | JSON数组 | 待询问问题列表，每个问题包含来源、类型、状态（pending/asked/confirmed/aborted） |
| `behavior_issues` | JSON数组 | 已确认的不良行为列表 |

**生命周期状态定义**：

*   **S1 新手启动期**：注册0-7天，数据空白，需引导
    
*   **S2 稳定执行期**：执行分≥80，体重呈预期下降
    
*   **S3 平台期风险**：体重连续7天无变化（±0.2kg），执行分≥70
    
*   **S4 风险期**：掉队（连续未打卡、执行分<50）、反弹（体重上升）、投诉风险等
    
*   **S5 稳态维持期**：达成阶段目标，维持2周，进入长期保持

### 三、主状态机（服务方案对象）详细转换表

服务方案为单一进行时，不会并行执行，如：免费尝试升付费，当前免费方案被替代，新方案付费

| 当前状态 | 事件 | 条件 | 下一状态 | 动作 |
| --- | --- | --- | --- | --- |
| **（无方案）** | 初评完成（first） | 可提供服务 | PENDING | 创建P1方案实例，发布`plan.pending` |
| **PENDING** | 阶段测评完成（phase） | \- | READY | 生成方案内容，发布`plan.generated` |
| **READY** | 用户点击“开始执行”（免费） | \- | ACTIVE | 设置`execution_start_date=激活次日`，发布`plan.activated`，次日0点生成任务（激活日为Day0，次日为执行日Day1） |
| **READY** | 用户点击“立即支付”（付费） | \- | AWAITING\_PAYMENT | 创建支付订单（CREATED），发布`payment.pending` |
| **AWAITING\_PAYMENT** | 支付成功 | \- | AWAITING\_DELIVERY | 订单状态为PAID，触发物流发货 |
| **AWAITING\_PAYMENT** | 支付失败 | \- | AWAITING\_PAYMENT | 记录失败，可重试 |
| **AWAITING\_PAYMENT** | 支付超时 | \- | READY | 订单EXPIRED，方案退回READY，需再次唤起支付 |
| **AWAITING\_DELIVERY** | 用户签收 | \- | ACTIVE | 激活方案（Day0），生成次日任务（次日为Day1） |
| **ACTIVE** | 周复评完成（weekly, 非第28天） | 达标 | ADJUSTED→ACTIVE | 微调任务（`fine_tune`），发布`plan.adjusted` |
| **ACTIVE** | 周复评完成（非第28天&非14天） | 不达标且执行分高 | ADJUSTED→ACTIVE | 强度升级（`intensify`），发布`plan.adjusted` |
| **ACTIVE** | 周复评完成（非第28天&非14天） | 不达标且执行分低（免费且用户升级） | （新付费方案） | 旧方案SUPERSEDED，新付费方案READY |
| **ACTIVE** | 周复评完成（非第28天&非14天） | 不达标且执行分低（免费但用户继续免费） | （无变化） | 创建干预流程，方案状态不变 |
| **ACTIVE** | 周复评完成（非第28天&非14天） | 不达标且执行分低（付费） | （无变化） | 创建干预流程 |
| **ACTIVE** | 周复评完成（第28天, is\_phase\_end=true） | 累计达标 | COMPLETED | 发布`plan.completed`，创建下一周期阶段测评 |
| **ACTIVE** | 周复评完成（第28天） | 累计不达标且`is_compliant=true` | TERMINATED | 发布`plan.terminated`（退款），用户terminated |
| **ACTIVE** | 周复评完成（第28天） | 累计不达标且`is_compliant=false` | TERMINATED | 发布`plan.terminated`（不退），用户terminated |
| **ACTIVE** | 第14天周复评不达标 | 提前终止规则（付费/ 免费） | TERMINATED | 同第28天处理 |
| **ACTIVE** | 用户主动升级（同周期） | \- | （新付费方案） | 旧方案SUPERSEDED，新付费方案READY |
| **COMPLETED** | 方案达标完结 | \- | 终态 | \- |
| **SUPERSEDED** | 方案被替代 | \- | 终态 | \- |
| **TERMINATED** | 用户点击“重新开始” | 用户态terminated | 触发新初评/阶段测评（P2）？ | 创建first测评/P2的阶段测评？ |

### 四、关键业务流程事件协同实例

#### 4.1 初评流程

1.  **注册**：创建用户（`user.active`），创建初评测评（`type=first`，`NOT_STARTED`），首页显示“开始测评”卡片。
    
2.  **进入测评**：用户点击 → 状态 （`IN_PROGRESS` ），进入作答。
    
    *   中途退出：用户点击退出或表达退出意图 → 弹出确认卡，选择“退出”则状态 `INTERRUPTED`，首页恢复卡片；选择“继续”则继续。
        
    *   答非所问：AI引导，最多2次，第三次提供退出选择，点退出（`INTERRUPTED`）/继续测评。
        
    *   不可服务：用户作答触发不可服务的回复内容，用户 `user.terminated`，测评状态`CANCELED`，首页出测评卡，可“重新测评”（新的初评）。
        
3.  **完成提交**：
    
    *   可服务：报告底部“获取方案”按钮。点击后创建P1方案（`PENDING`）和阶段测评（`type=phase`，`IN_PROGRESS`）。
        
4.  **阶段测评**：类似初评流程（可中断/取消而退出），完成后发布事件，方案变为 `READY`，前端展示免费/付费方案切换。
    
5.  **用户选择**：
    
    *   免费：点击“开始执行” → `ACTIVE`，次日生成任务。
        
    *   付费：点击“立即支付” → `AWAITING_PAYMENT`，支付成功后进入 `AWAITING_DELIVERY`，签收后 `ACTIVE`。
        

#### 4.2 周复评流程（非第28天）

1.  **触发**：第7/14/21天定时创建周复评测评（`type=weekly`，`is_phase_end=false`），首页显示“本周复评”卡片。
    
2.  **用户完成**：提交后发布事件（含达标、归因、微调建议、执行分）。
    
3.  **方案处理**：
    
    *   达标：微调任务（`ADJUSTED`_last\_adjustment\_type_\=_fine\_tune_ →`ACTIVE`）。
        
    *   非14天不达标且执行分高：强度升级（`ADJUSTED`_last\_adjustment\_type_\=_intensify_ →`ACTIVE`）。
        
    *   非14天不达标且执行分低（免费）：出引导选择升级或继续免费。
        
        *   升级：旧方案 `SUPERSEDED`，新付费方案 `READY`。
            
        *   继续免费：创建干预流程（干预方向待定，由干预流程处理DETECTED）。
            
    *   非14天不达标且执行分低（付费）：创建干预流程（干预方向待定，由干预流程处理DETECTED）。
        
    *   14天不达标（付费/ 免费）：方案 `TERMINATED`，用户 `terminated`，同28天不达标处理。
        

#### 4.3 第28天复评流程

1.  **触发**：第28天创建周复评测评（`type=weekly`，`is_phase_end=true`）。
    
2.  **完成**：发布事件（含是否达标、归因 `is_compliant`）。
    
3.  **方案处理**：
    
    *   达标：当前方案 `COMPLETED`，创建下一周期阶段测评（type=phase），用户完成得新周期方案。
        
    *   不达标：方案 `TERMINATED`，根据归因决定退款，用户 `terminated`，展示终止页及“重新开始”。
        

#### 4.4 执行与反馈流程（打卡、追问、放弃）

1.  **每日任务生成**：0点生成当日任务，每任务创建 `PENDING`，`due_time=23:59`，`feedback_status=pending`。
    
2.  **用户打卡**：
    
    *   有子任务，点击单一子任务完成 → 子任务 `COMPLETED`；当所有主任务中子任务完成，驱动总任务 `COMPLETED`，发布事件，更新执行分。
        
    *   有子任务，点击批量完成任务 → 所有子任务 `COMPLETED`，驱动总任务 `COMPLETED`，发布事件，更新执行分。
        
    *   无子任务，点击完成 → 任务 `COMPLETED`，发布事件，更新执行分。
        
3.  **触发反馈**：若 `feedback_status=pending`，改为 `in_progress`，`feedback_attempts=1`，`expire_time=now+1h`，聊天区发起追问，首页顶部显示反馈状态栏 + “退出”按钮。
    
4.  **交互**：
    
    *   有效回答：存入 `feedback_data`，状态 `completed`，状态栏消失。
        
    *   无效回答：再次追问，`attempts++`；≥1次无效则放弃，反馈状态 `aborted`（数据空），状态栏消失。
        
    *   点击“退出”：放弃，反馈状态 `aborted`，同时生成待询问问题加入 `pending_questions`（是否还问?不再询问）。
        
    *   超时（1h）：自动放弃，反馈状态 `aborted`。
        
5.  **后续**：收集数据更新分数和画像。
    

#### 4.5 日常问询流程

1.  **触发条件**：用户打开首页，无更高优先级事项（无进行中测评/打卡/干预/反馈），`pending_questions` 非空，距上次日常问询>24h。
    
2.  **发起**：从列表中选一个问题，聊天区询问（无顶部状态栏是否要加状态？可隐性）。
    
3.  **回复处理**：
    
    *   有效回答：更新数据/确认不良行为，已确认问题移除。
        
    *   模糊回答：追问1-2次，仍无法确认则标记  `aborted`。
        
    *   用户主动开启新话题：暂停当前问询（状态 `asked`，如何处理？AI自己识别），优先回应用户。
        
4.  **超时**：1小时无响应则 `aborted`。
    

#### 4.6 监督与干预流程

1.  **监督模块**：监听事件，实时计算执行分、能量分、风险分，更新生命周期。
    
    *   执行期：打卡任务内容、打卡后反馈、用户主聊反馈，日常问询后收集、任务超时等。
        
    *   非28天复评：收集问卷，特定 数据/选项触发
        
    *   监督模块监听的事件源如下
        
    
    | 事件类型 | 具体事件 | 来源对象 | 触发时机 |
    | --- | --- | --- | --- |
    | **执行期事件** | `task.completed`（任务打卡） | 每日任务 | 用户打卡时 |
    |  | `task.overdue`（任务超时） | 每日任务 | 定时任务检测到超时 |
    |  | `task.feedback_submitted`（打卡后反馈完成） | 每日任务 | 反馈追问完成时 |
    |  | `chat.negative_sentiment`（主聊负面情绪） | AI对话 | 实时分析用户消息 |
    |  | `device.data_synced`（设备数据同步） | 硬件 | 用户同步体重/体脂时 |
    |  | `daily_inquiry.completed`（日常问询完成） | 日常问询 | 日常问询对话结束时 |
    | **测评期事件** | `assessment.completed`（测评完成） | 测评流程 | 用户提交测评问卷 |
    |  | `assessment.risk_answer`（测评中高风险回答） | 测评流程 | 用户回答特定敏感选项时立即触发 |
    | **用户主动求助** | `user.help_requested` | 前端 | 用户点击“求助”按钮或在聊天中明确求助 |
    | **系统定时任务** | 每日分数衰减、连续未打卡检测 | 定时任务 | 每天固定时间 |
    
2.  **干预触发**：达到阈值（如风险分>60、执行分<50持续3天等），创建干预流程（`DETECTED`）。
    
3.  **干预决策**：
    
    *   L1（`REMINDING`）：发送Push/聊天提醒，24小时无响应则升级。
        
    *   L2（`ADJUSTING`）：推送Plan B任务，24小时无响应则升级。
        
    *   L3（`HUMAN`）：转人工顾问。
        
4.  **用户响应**：完成相关任务 → 干预 `RESOLVED`。
    
5.  **多次无响应（升级）**：`ESCALATED` → 重新决策。

### 五、分数体系与AI决策

#### 5.1 分数定义与计算规则

| 分数 | 含义 | 初始值 | 更新频率 | 计算要素与调整规则 |
| --- | --- | --- | --- | --- |
| **执行分** | 任务完成质量与依从性 | 60 | 每日 | 加分：每个任务按时完成+2，提前完成+1；连续3天全完成+5。<br>扣分：每个任务超时完成-1（仍完成），未完成-5；连续3天未完成-10。<br>上限100，下限0。 |
| **能量分** | 用户身心承受力 | 70 | 每日 | 加分：睡眠≥7h+5，疲劳自评≤2+5，运动后反馈积极+3，对话情绪积极+2。<br>扣分：睡眠<5h-10，疲劳自评≥4-5，运动后反馈消极-3，对话情绪消极-2。<br>每日变化累加，上限100，下限0。 |
| **风险分** | 安全红线与流失概率 | 0 | 实时/每日 | 触发事件加分（取最大值，不累加）：<br>\- 连续未打卡：每1天+10（累计3天达30）<br>\- 体重反弹：每0.1kg+5<br>\- 医疗边界：+100<br>\- 极端减重：+50<br>\- 投诉词：+50<br>衰减：每日无新事件时，风险分衰减10%（向下取整），最低0。 |

#### 5.2 生命周期更新规则

| 条件 | 生命周期 |
| --- | --- |
| 注册0-7天，执行分<60 | S1新手启动期 |
| 执行分≥80且体重下降（7天均值下降） | S2稳定执行期 |
| 体重连续7天无变化（±0.2kg）且执行分≥70 | S3平台期风险 |
| 连续3天未打卡/执行分<50/体重上升（7天均值上升）/风险分≥50，任一满足 | S4风险期 |
| 达成阶段目标，维持2周（执行分≥80，体重稳定±0.2kg） | S5稳态维持期 |

#### 5.3 干预决策规则

| 条件 | 干预等级 |
| --- | --- |
| 风险分<60，执行分50-70 | L1提醒 |
| 风险分60-80，执行分<50，或2次L1无效 | L2调整 |
| 风险分>80，或两次L2无效，或医疗边界 | L3人工 |

### 八、主聊天通道调度策略

**优先级**（从高到低）：

| 优先级 | 对话类型 | 界面状态栏 |
| --- | --- | --- |
| 1 | 用户主动消息 | 无（用户主导） |
| 2 | 测评（初评/复评/阶段） | 有（进度条+退出） |
| 3 | 干预提醒 | 有（可定制） |
| 4 | 打卡后反馈 | 有（反馈+退出） |
| 5 | 日常问询 | 无 |

**调度规则**：

*   当高优先级对话活跃时，低优先级不得发起。
    
*   用户主动消息打断当前对话：保存进度（测评中断、反馈放弃生成待问询），回复用户后不再自动恢复。
    
*   各对话有超时/放弃机制，防止长期占用。

本设计以服务方案对象为主脉络，通过多对象状态机协同，完整覆盖了初评、周复评、第28天复评、执行反馈、日常问询、监督干预、付费升级降级、终止退款等核心业务场景。各对象状态清晰，事件驱动，调度合理，易于扩展。各团队可基于此骨架并行开发，确保产品高效落地。
