## 1. 文档目的

本文档定义好享健康 APP 中"监督与调整"模块的完整产品方案，供产品、前端、后端、AI、数据、测试、顾问团队评审与落地。

本模块不是单纯的提醒系统，而是 AI Agent 驱动健康管理的控制层，负责在全生命周期内持续感知用户状态，识别信号，生成动作，并验证动作是否生效。

### 1.1 覆盖阶段

| 阶段 | 说明 |
|------|------|
| 首次测评 | 用户初次使用时的测评流程 |
| 报告后继续获取方案 | 测评报告生成后的方案获取 |
| 方案生成与待激活 | 方案确认与激活 |
| 付费待支付、待发货、待签收 | 订单履约阶段 |
| Day0 与 Day1 | 服务启动阶段 |
| 28 天服务执行期 | 方案执行阶段 |
| Day7/14/21 周复评 | 周期性复评 |
| Day28 阶段复评 | 阶段总结复评 |
| 下阶段承接 | 阶段切换 |
| 服务终止与退款评估 | 服务结束 |

## 2. 模块定义与核心职责

### 2.1 核心职责

监督与调整模块的核心职责包括：

| 职责 | 说明 |
|------|------|
| **持续监督** | 覆盖用户全生命周期的状态监控 |
| **信号识别** | 从显性/隐性事件中提取有效信号 |
| **分数更新** | 执行分、能量分、风险分的动态计算 |
| **生命周期切换** | S1-S5 阶段的准确判断与转换 |
| **调整策略生成** | 根据分数和信号生成干预动作 |
| **触达与执行** | 通过合适渠道触达用户 |
| **结果回写** | 记录干预效果，形成闭环 |

### 2.2 统一主链路

```
监督来源 → 事件 → 信号 → 分数 → 生命周期 → 干预等级 → 调整动作 → 回写 → 再观察
```

## 4. 核心设计原则

### 4.1 全生命周期持续监督

监督不是只发生在执行期，而是贯穿以下所有阶段：

| 阶段类型 | 说明 |
|----------|------|
| 测评前和测评中 | 首次使用引导 |
| 方案待获取与待激活 | 转化阶段 |
| 履约等待期 | 订单处理 |
| Day0、Day1 | 启动阶段 |
| 执行期 | 28天服务 |
| 复评期 | 周复评/阶段复评 |
| 阶段切换期 | 阶段流转 |
| 终止和退款评估期 | 服务结束 |

### 4.2 显性与隐性并行

监督必须同时覆盖两类信息：

| 类型 | 说明 | 示例 |
|------|------|------|
| **显性监督** | 用户明确完成、反馈、表达 | 打卡完成、主诉反馈 |
| **隐性监督** | 不打卡、不回复、临近截止才完成 | 连续不打卡、打开不执行 |

### 4.3 规则、AI、混合判定分层

| 判定方式 | 适用场景 |
|----------|----------|
| **规则引擎** | 明确阈值、明确时点、必须稳定的判断 |
| **AI 语义** | 自然语言理解、阻碍识别、语义归因 |
| **混合判定** | AI结构化提取后由规则引擎落动作 |

### 4.4 风险优先级最高

任何阶段都遵循以下优先级：

| 优先级 | 说明 |
|--------|------|
| P1 | 医疗和安全边界 |
| P2 | 正在进行中的测评和复评 |
| P3 | 当前主任务闭环 |
| P4 | 风险纠偏 |
| P5 | 日常主动问询 |
| P6 | 商业转化和续费 |

### 4.5 调整必须回写和验证

系统必须记录：

| 记录项 | 说明 |
|--------|------|
| 为什么调 | 调整原因 |
| 调了什么 | 调整内容 |
| 用户是否接受 | 用户确认状态 |
| 调整后是否改善 | 效果验证 |
| 是否继续/恢复/升级/终止 | 后续动作 |

## 5. 决策思维链

监督与调整模块的决策必须按以下 8 步执行：

### 5.1 决策流程

```
┌─────────────────────────────────────────────────────────────────────┐
│                        决策思维链 8 步                               │
├─────────────────────────────────────────────────────────────────────┤
│  Step 1: 识别当前所处阶段                                           │
│           ↓                                                         │
│  Step 2: 收集监督事件（显性 + 隐性）                               │
│           ↓                                                         │
│  Step 3: 生成信号（归并 + 解释）                                   │
│           ↓                                                         │
│  Step 4: 更新分数（执行分 + 能量分 + 风险分）                      │
│           ↓                                                         │
│  Step 5: 判断生命周期（S1-S5）                                     │
│           ↓                                                         │
│  Step 6: 判定干预等级（L0-L3）                                     │
│           ↓                                                         │
│  Step 7: 选择动作和触达渠道                                        │
│           ↓                                                         │
│  Step 8: 回写与效果验证                                            │
└─────────────────────────────────────────────────────────────────────┘
```

### 5.2 各步骤详细说明

| 步骤 | 输入 | 输出 |
|------|------|------|
| Step 1 | 用户状态数据 | 当前阶段判定 |
| Step 2 | 显性事件 + 隐性事件 | 事件列表 |
| Step 3 | 事件列表 | 信号列表 |
| Step 4 | 信号 + 历史分数 | 新分数 |
| Step 5 | 分数 + 趋势 + 目标 | 生命周期 |
| Step 6 | 分数 + 信号 | 干预等级 |
| Step 7 | 干预等级 + 用户偏好 | 动作 + 渠道 |
| Step 8 | 动作执行结果 | 效果评估 |

## 7. 统一对象模型与分数体系

### 7.1 核心状态对象

| 对象类型 | 说明 |
|----------|------|
| 服务方案对象 | 用户购买的服务方案 |
| 测评对象 | 首次/周/阶段测评 |

| 订单与履约对象 | 付费/发货/签收 |
| 用户生命周期对象 | S1-S5 阶段 |

### 7.2 三大分数体系

#### 7.2.1 执行分（execution_score）

| 属性 | 值 |
|------|-----|
| 默认值 | 60 |

## 8. 生命周期定义

### 8.1 生命周期总览

| 阶段 | 名称 | 条件 | 重点 |
|------|------|------|------|
| S1 | 新手启动期 | 注册后到方案激活后前7天 | 补档、Day0/Day1引导、降低摩擦 |
| S2 | 稳定执行期 | 执行分≥65，风险分<30 | 稳定执行、适度升级、强化反馈 |
| S3 | 平台期风险 | 执行分≥60，真实平台期 | 平台期解释、饮食微调、防流失 |
| S4 | 风险期 | 风险分≥50 或 执行分<45 | 拉回、缩包、Plan B、必要人工 |
| S5 | 稳态维持期 | 达到阶段目标且稳定7-14天 | 频率下调、防反弹、维持习惯 |

### 8.2 S1 新手启动期

| 项目 | 内容 |
|------|------|
| 条件 | 注册后到方案激活后的前7天，或关键数据尚未完整 |
| 重点 | 补档、Day0/Day1引导、降低摩擦 |

### 8.3 S2 稳定执行期

| 项目 | 内容 |
|------|------|
| 条件 | execution_score ≥ 65，risk_score < 30，无平台期和高危风险 |
| 重点 | 稳定执行、适度升级、强化反馈 |

### 8.4 S3 平台期风险

| 项目 | 内容 |
|------|------|
| 条件 | 真实平台期成立，execution_score ≥ 60 |
| 重点 | 平台期解释、饮食方法微调、防焦虑和流失 |

### 8.5 S4 风险期

满足任一条件即进入：

| 条件 | 说明 |
|------|------|
| risk_score ≥ 50 | 风险分达到阈值 |
| execution_score < 45 | 执行分过低 |
| 连续3天无核心任务完成 | 任务未完成 |
| 信任危机或高危表达 | 风险表达 |

重点：拉回、缩包、Plan B、必要时人工

### 8.6 S5 稳态维持期

| 项目 | 内容 |
|------|------|
| 条件 | 达到阶段目标，稳定维持7到14天 |
| 重点 | 频率下调、防反弹、维持习惯 |

## 15. 行为干预任务 PRD

### 15.1 任务定位

行为干预是"今日三件事"中的第3个任务。

| 任务来源 | 类型 |
|----------|------|
| 行为库 | 数据追踪类 / 行为习惯类 |

### 15.2 行为追问四个评分维度

| 维度 | 权重 | 说明 |
|------|------|------|
| 真实性 | 30% | 用户是否真实执行 |
| 执行质量 | 25% | 执行的程度如何 |
| 效果感知 | 20% | 用户是否感受到效果 |
| 可持续性 | 25% | 是否容易坚持 |

**建议公式：**

```
behavior_score = 0.30*真实性 + 0.25*执行质量 + 0.20*效果感知 + 0.25*可持续性
```

### 15.3 Day1/Day2/Day3 目标

| 天数 | 目标 | 获取维度 |
|------|------|----------|
| Day1 | 确认是否真实执行 | 完成度 |
| Day2 | 判断执行质量、是否开始产生效果 | 执行质量、效果感知 |
| Day3 | 判断是否可形成习惯 | 可持续性 |

### 15.4 行为任务可选动作

| 动作 | 说明 |
|------|------|
| 继续当前任务 | 保持现有任务 |
| 难度调整 | 参数调整 |
| 行为替换 | 更换任务 |
| 任务移除 | 移除任务 |
| 完成并流转 | 进入下一任务 |

### 15.5 3天评估动作矩阵

| 完成情况 | 评分 | 动作 |
|----------|------|------|
| 0/3或1/3完成，无回复 | - | 隐性掉队，标签保留，降入P3 |
| 0/3或1/3完成，用户明确说难 | 核心动作合理 | 先难度调整 |
| 0/3或1/3完成，用户明确说难 | 核心动作不合理 | 行为替换 |
| 0/3或1/3完成，用户明确拒绝 | - | 任务移除 |
| 2/3完成，Day3失败 | - | 优先难度调整 |
| 3天完成 | ≥4.2 | 继续巩固/流转下一任务，标签weakened |
| 3天完成 | 3.4-4.2 | 继续当前任务，不升级难度 |
| 3天完成 | 2.6-3.4 | 优先难度调整 |
| 3天完成 | 1.8-2.6 | 优先行为替换，新任务进P1顶部 |
| 3天完成 | <1.8 | 任务移除，次日切换下一任务，标签保留 |

### 15.6 难度调整规则

难度调整是在不改变目标和核心动作的前提下调参数：

| 参数 | 示例 |
|------|------|
| 次数 | 3次 → 1次 |
| 频次 | 每天 → 隔天 |
| 时长 | 10分钟 → 3分钟 |
| 执行窗口 | 睡前1小时 → 晚饭后30分钟内 |
| 执行场景 | 随时 → 特定场景 |

### 15.7 行为替换规则

行为替换不是从行为库直接选一个旧任务，而是AI基于原任务目标生成一个更好执行的新任务。

| 约束 | 说明 |
|------|------|
| 目标不变 | 保持健康目标 |
| 成本更低 | 降低执行门槛 |
| 更适配当前场景 | 匹配用户实际情况 |

### 15.8 任务移除规则

以下情况可移除：

| 情况 | 说明 |
|------|------|
| Day1明确反馈不合适 | 用户说太简单/没必要/不适合 |
| 连续不打卡且不回应 | 隐性放弃 |
| 评分极低 | behavior_score < 1.8 |
| 用户明确拒绝 | 直接拒绝 |

## 16. 应用维度分数映射到行为干预

### 16.1 设计背景

原系统仅通过四个维度（真实性、执行质量、效果感知、可持续性）的行为评分驱动行为干预任务调整。为实现更全面的用户状态感知，现引入应用维度的"能量分、执行分、风险分"作为第二驱动源。

### 16.2 双分数驱动架构

```
┌─────────────────────────────────────────────────────────────────────┐
│                    行为干预任务调整双分数驱动                        │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   分数驱动源1：行为评分                                            │
│   ┌─────────────────────────────────────────────────────────────┐  │
│   │ 真实性(30%) + 执行质量(25%) + 效果感知(20%) + 可持续性(25%)  │  │
│   │                    ↓                                         │  │
│   │              behavior_score (1-5分)                          │  │
│   └─────────────────────────────────────────────────────────────┘  │
│                              ↓                                      │
│   分数驱动源2：应用维度分数                                         │
│   ┌─────────────────────────────────────────────────────────────┐  │
│   │  execution_score (执行分) ←→ 能量分/执行分/风险分            │  │
│   │  energy_score (能量分)     ←→                              │  │
│   │  risk_score (风险分)       ←→                               │  │
│   └─────────────────────────────────────────────────────────────┘  │
│                              ↓                                      │
│   综合决策引擎                                                      │
│   ┌─────────────────────────────────────────────────────────────┐  │
│   │  行为评分 × 应用维度分数 = 综合干预决策                        │  │
│   └─────────────────────────────────────────────────────────────┘  │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 16.3 应用维度分数对行为干预的映射规则

#### 16.3.1 能量分（energy_score）对行为干预的影响

| 能量分范围 | 行为任务调整策略 |
|------------|------------------|
| ≥80 | 可提升行为任务难度，维持正常节奏 |
| 65-79 | 维持当前行为任务难度 |
| 50-64 | 降低行为任务频次/时长 |
| 35-49 | 替换为更轻松的行为任务，简化目标 |
| <35 | 暂停行为干预任务，优先恢复能量 |

**影响权重：40%**

#### 16.3.2 执行分（execution_score）对行为干预的影响

| 执行分范围 | 行为任务调整策略 |
|------------|------------------|
| ≥70 | 任务完成良好，可升级挑战 |
| 55-69 | 维持当前任务难度 |
| 45-54 | 降低任务难度，缩小任务包 |
| 30-44 | 简化任务，或替换为更易执行的任务 |
| <30 | 暂停行为干预，分析掉队原因 |

**影响权重：40%**

#### 16.3.3 风险分（risk_score）对行为干预的影响

| 风险分范围 | 行为任务调整策略 |
|------------|------------------|
| <15 | 正常执行行为任务 |
| 15-29 | 关注行为完成情况，轻度干预 |
| 30-49 | 降低行为任务要求，优先完成 |
| 50-79 | 简化行为任务，缩短周期 |
| ≥80 | 暂停行为干预，升级L3人工介入 |

**影响权重：20%**

### 16.4 综合评分计算

#### 16.4.1 应用维度综合行为分数

```
app_behavior_score = 
    (execution_score * 0.4) + 
    (energy_score * 0.4) + 
    ((100 - risk_score) * 0.2)
```

#### 16.4.2 最终行为干预决策

```
final_decision = 
    f(behavior_score, app_behavior_score, lifecycle_stage)
```

**决策矩阵：**

| 行为评分 | 应用维度评分 | 建议动作 |
|----------|--------------|----------|
| 高(≥4) | 高(≥70) | 继续当前任务/升级难度 |
| 高(≥4) | 中(50-70) | 维持当前任务 |
| 高(≥4) | 低(<50) | 降低难度/关注能量 |
| 中(2.5-4) | 高(≥70) | 维持或小幅调整 |
| 中(2.5-4) | 中(50-70) | 难度调整 |
| 中(2.5-4) | 低(<50) | 简化任务/替换 |
| 低(<2.5) | 高(≥70) | 分析原因/替换任务 |
| 低(<2.5) | 中(50-70) | 简化任务/降低预期 |
| 低(<2.5) | 低(<50) | 暂停/人工介入 |

### 16.5 触发规则

#### 16.5.1 触发时机

| 触发时机 | 说明 |
|----------|------|
| 每日分数结算时 | 每日00:10分数更新后 |
| 行为任务完成打卡时 | 每次打卡后评估 |
| 用户反馈困难时 | 实时触发 |
| 周复评时 | 周期性评估 |

#### 16.5.2 评估频率

| 评估点 | 频率 | 说明 |
|--------|------|------|
| 每日结算 | 每日1次 | 综合评估 |
| 打卡后评估 | 实时 | 单次任务评估 |
| 周复评评估 | 每周1次 | 周期评估 |

### 16.6 动作输出

#### 16.6.1 任务调整动作库

| 动作类型 | 触发条件 | 具体动作 |
|----------|----------|----------|
| 继续当前任务 | 评分良好 | 保持不变 |
| 难度提升 | 高评分持续3天 | 增加频次/时长 |
| 难度降低 | 中低评分 | 减少参数 |
| 任务替换 | 持续低评分 | AI生成新任务 |
| 任务暂停 | 极低评分/风险高 | 暂停执行 |
| 人工介入 | 风险分≥80 | 升级L3 |

#### 16.6.2 反馈话术模板

| 场景 | AI话术 |
|------|--------|
| 能量低 | "最近感觉比较疲惫吧？我们先把行为目标降低一些，养足精神更重要～" |
| 执行低 | "这个任务对你来说可能有难度，我们换一个更容易坚持的方式吧？" |
| 风险高 | "我注意到你最近压力比较大，需要我帮你联系人工顾问吗？" |

| 短信发送失败 | 记录日志，转入人工处理 |
| 用户手机号无效 | 标记，转入人工召回 |
| 超过发送频率限制 | 加入队列，等待发送 |

## 18. 方案调整规则

### 18.1 周期内微调

28天服务周期内允许的主要调整：

| 可调整 | 不可调整 |
|--------|----------|
| 饮食方法 | 服务阶段本身 |
| 饮食任务参数 | 不因单次波动切换下一阶段 |
| 运动任务难度/频次/时长 | 不做大阶段重生成 |
| 行为任务难度和顺序 | - |
| 提醒频率 | - |

### 18.2 周复评后调整

周复评后可做：

| 可调整 | 不可调整 |
|--------|----------|
| 饮食方法微调 | 服务阶段切换 |
| 任务参数微调 | 下一阶段方案生成 |
| 行为优先级调整 | - |
| 运动强度轻调 | - |

### 18.3 阶段复评后调整

#### 18.3.1 达标处理

| 条件 | 处理 |
|------|------|
| 达标 | 当前阶段完成，进入下一阶段 |
| 免费方案 | 确认后次日Day1 |
| 付费方案 | 支付签收后次日Day1 |

#### 18.3.2 未达标处理

未达标不直接进入下一阶段，必须先归因：

| 归因 | 说明 |
|------|------|
| A1 | 执行不足 |
| A2 | 饮食方法不匹配 |
| A3 | 能量或生理波动 |
| A4 | 场景干扰过强 |
| A5 | 产品或履约问题 |

## 19. 标签流转规则

### 19.1 标签状态

| 状态 | 说明 |
|------|------|
| suspected | 疑似 |
| confirmed | 确认 |
| intervening | 干预中 |
| weakened | 弱化 |
| resolved | 已解决 |

### 19.2 标签进入

| 进入条件 | 说明 |
|----------|------|
| 测评命中 | 测评问卷识别 |
| 主动问询承认 | 用户主动说明 |
| AI语义识别 | 明确主诉 |
| 隐性信号重复 | 出现≥2次 |

### 19.3 标签升级

| 升级条件 | 说明 |
|----------|------|
| 7天内同类问题重复 | 反复出现 |
| 当前行为任务失败 | 任务失败 |
| 隐性和显性同时命中 | 多重确认 |

### 19.4 标签弱化

满足任一即可：

| 条件 | 说明 |
|------|------|
| 1个周期 behavior_score≥4.2 | 表现良好 |
| 7天内无重复负向信号 | 无新增问题 |

### 19.5 标签解决

满足以下组合：

| 条件 | 说明 |
|------|------|
| 2个相关任务周期表现良好 | 连续良好 |
| 连续14天无相关信号 | 长期稳定 |
| 若有客观数据，不恶化 | 数据支持 |

### 19.6 标签保留

以下情况标签保留：

| 状态 | 说明 |
|------|------|
| 任务未完成 | 进行中 |
| 行为评分低 | 不达标 |
| 用户明确拒绝 | 不配合 |
| 隐性负向信号仍持续 | 风险持续 |

| Day28复评 | +48h未完成 | L2 |
| 发货成功 | - | 即时Push |

## 23. 验收标准

### 23.1 监督有效

- [ ] 系统能识别显性与隐性监督事件
- [ ] 体重、签收等关键事件实时优先
- [ ] 定时检测只做缺失和兜底

### 23.2 判定可解释

- [ ] 能说清楚某次动作为什么触发
- [ ] 能回放 event → signal → score → action

### 23.3 动作可执行

- [ ] 任务调整能落到实际任务
- [ ] 行为替换能生成新任务并进入P1
- [ ] 终止和退款评估有明确入口

### 23.4 渠道不打扰

- [ ] Push有冷却和优先级
- [ ] 高风险优先，普通提醒克制

### 23.5 全生命周期打通

- [ ] 首次测评、方案、执行、复评、阶段流转、终止全部被覆盖
