## 1. 文档目的

本文档定义好享健康 APP 中"监督与调整"模块的完整产品方案，供产品、前端、后端、AI、数据、测试、顾问团队评审与落地。

本模块不是单纯的提醒系统，而是 AI Agent 驱动健康管理的控制层，负责在全生命周期内持续感知用户状态，识别信号，生成动作，并验证动作是否生效。

**本模块覆盖阶段**：
- 首次测评
- 报告后继续获取方案
- 方案生成与待激活
- 付费待支付、待发货、待签收
- Day0 与 Day1
- 28 天服务执行期
- Day7/14/21 周复评
- Day28 阶段复评
- 下阶段承接
- 服务终止与退款评估

## 2. 模块定义

监督与调整模块的核心职责是：

1. **持续监督** - 贯穿用户全生命周期
2. **信号识别** - 从事件中提取有效信号
3. **分数更新** - 维护三大核心分数
4. **生命周期切换** - 智能判断用户所处阶段
5. **调整策略生成** - 基于信号和分数生成动作
6. **触达与执行** - 选择合适渠道触达用户
7. **结果回写** - 记录并验证调整效果

**统一主链路**：

```
监督来源 → 事件 → 信号 → 分数 → 生命周期 → 干预等级 → 调整动作 → 回写 → 再观察
```

## 3. 模块目标

### 3.1 用户目标
- 用户始终知道当前最重要的下一步
- 用户在困难时能被及时理解和拉回
- 用户不会因平台期、低能量、场景波动被误判
- 用户在高风险时能被及时保护和升级人工

### 3.2 业务目标
- 提升测评完成率
- 提升方案激活率
- 提升 Day0 与 Day1 完成率
- 提升今日三件事执行率
- 提升周复评与阶段复评完成率
- 降低掉队率、投诉率、无反馈率

### 3.3 系统目标
- 可解释
- 可追溯
- 可配置
- 可灰度
- 可审计

## 4. 核心设计原则

### 4.1 全生命周期持续监督
监督不是只发生在执行期，而是贯穿以下所有阶段：
- 测评前和测评中
- 方案待获取与待激活
- 履约等待期
- Day0、Day1
- 执行期
- 复评期
- 阶段切换期
- 终止和退款评估期

### 4.2 显性与隐性并行
监督必须同时覆盖两类信息：
- **显性监督**：用户明确完成、明确反馈、明确表达
- **隐性监督**：不打卡、不回复、临近截止才完成、打开不执行、短回复、重复同一困难

### 4.3 规则、AI、混合判定分层
- **规则引擎**：负责明确阈值、明确时点、必须稳定的判断
- **AI**：负责自然语言理解、阻碍识别、语义归因
- **混合判定**：负责 AI 结构化提取后由规则引擎落动作

### 4.4 风险优先级最高
任何阶段都遵循以下优先级：
1. 医疗和安全边界
2. 正在进行中的测评和复评
3. 当前主任务闭环
4. 风险纠偏
5. 日常主动问询
6. 商业转化和续费

### 4.5 调整必须回写和验证
系统不能只"调一次"，必须记录：
- 为什么调
- 调了什么
- 用户是否接受
- 调整后 3 天或 7 天是否改善
- 是否继续、恢复、升级或终止

## 5. 干预等级定义（前置说明）

> **重要说明**：本文档将干预等级定义前置，因为在第12点之前已经出现L1/L2/L3的概念，为了便于理解，先统一定义。

| 等级 | 名称 | 适用场景 | 动作类型 |
|------|------|----------|----------|
| **L0** | 不干预 | 风险低、执行稳定、当前无必要额外触达 | 无 |
| **L1** | 提醒 | 单次未完成、单次数据缺失、轻度拖延、初次复评超时 | 首页卡片、站内消息、AI轻提醒、Push |
| **L2** | 调整 | 连续偏差、低能量、平台期、多次提醒无效、行为任务明显不适配 | 缩小任务包、任务降级、替代任务、行为替换、饮食方法微调、周末模式、Plan B |
| **L3** | 人工 | 高危红线、医疗边界、投诉退费、连续多次L2无效、高价值用户长期沉默 | 顾问接管、工单、电话/企微、医学或营养师介入 |

**触发规则**：
- risk_score >= 80 → 直接 L3
- 高危红线和医疗边界不等待二次确认，直接升级L3

## 6. 决策思维链

监督与调整模块的决策必须按以下8步执行：

### Step 1：识别当前所处阶段
先判断用户当前在哪个阶段：
- 首次测评
- 报告后待继续
- 方案待激活
- 付费待支付
- 付费待签收
- Day0
- 执行期
- 周复评
- 阶段复评
- 下阶段承接
- 终止

**注意**：系统不能跨阶段误触发。例如：
- 正在复评时不能先发日常问询
- 待签收时不能提前进入付费 Day1

### Step 2：收集监督事件
从显性事件和隐性事件中收集当前有效监督输入。

### Step 3：生成信号
对事件做归并和解释，得到策略可用的信号，而不是直接根据单一事件下动作。

### Step 4：更新分数
更新：
- 执行分（execution_score）
- 能量分（energy_score）
- 风险分（risk_score）

### Step 5：判断生命周期
根据分数、趋势、目标达成情况更新：
- S1 新手启动期
- S2 稳定执行期
- S3 平台期风险
- S4 风险期
- S5 稳态维持期

### Step 6：判定干预等级
根据上述分数和信号，判定：
- L0 不干预
- L1 提醒
- L2 调整
- L3 人工

### Step 7：选择动作和触达渠道
动作包括：
- AI 解释
- 任务调整
- 行为难度调整
- 行为替换
- 任务移除
- 饮食方法微调
- 方案切换
- Push / 站内 / 首页 / 短信 / 人工

### Step 8：回写与效果验证
验证窗口建议按动作类型分层：
- 轻提醒：24 小时
- 任务调整：3 天
- 行为任务调整：3 天或 1 个周期
- 周复评后微调：7 天
- 人工干预：由人工 SLA 驱动

## 7. 三大分数体系

### 7.1 执行分（execution_score）

| 属性 | 值 |
|------|-----|
| 默认值 | 60 |
| 核心用途 | 判断是否在执行，驱动任务升级、降级、缩包，驱动 S1/S2/S4 |

### 7.2 能量分（energy_score）

| 属性 | 值 |
|------|-----|
| 默认值 | 70 |
| 核心用途 | 判断做不做得动，驱动运动和行为任务轻重变化，决定 AI 是否切换照护语气 |

### 7.3 风险分（risk_score）

| 属性 | 值 |
|------|-----|
| 默认值 | 0 |
| 核心用途 | 判断掉队、流失、投诉、安全风险，决定 L1/L2/L3，决定是否立即人工介入 |

## 8. 生命周期定义

### 8.1 S1 新手启动期

| 项目 | 内容 |
|------|------|
| **进入条件** | 注册后到方案激活后的前 7 天，或关键数据尚未完整 |
| **重点** | 补档、Day0/Day1 引导、降低摩擦 |
| **规则** | 执行分>=65 且 风险分<30 且 无平台期和高危风险 |

### 8.2 S2 稳定执行期

| 项目 | 内容 |
|------|------|
| **进入条件** | 执行分 >= 65，风险分 < 30，无平台期和高危风险 |
| **重点** | 稳定执行、适度升级、强化反馈 |

### 8.3 S3 平台期风险

| 项目 | 内容 |
|------|------|
| **进入条件** | 真实平台期成立，执行分 >= 60（见第9节平台期判定规则） |
| **重点** | 平台期解释、饮食方法微调、防焦虑和流失 |

### 8.4 S4 风险期

| 项目 | 内容 |
|------|------|
| **进入条件** | 满足任一：风险分 >= 50；执行分 < 45；连续 3 天无核心任务完成；信任危机或高危表达 |
| **重点** | 拉回、缩包、Plan B、必要时人工 |

### 8.5 S5 稳态维持期

| 项目 | 内容 |
|------|------|
| **进入条件** | 达到阶段目标，稳定维持 7 到 14 天 |
| **重点** | 频率下调、防反弹、维持习惯 |

## 9. 平台期判定规则

### 9.1 真实平台期（True Plateau）

**判定条件**（需同时满足）：

| 条件 | 阈值 | 说明 |
|------|------|------|
| 连续体重数据 | ≥7天 | 每日有体重记录 |
| 体重波动 | <0.2kg | 7天平均体重与前一天差值 |
| 执行分 | ≥60 | 任务执行正常 |
| 能量分 | ≥60 | 用户能量状态正常 |
| 无明显阻碍主诉 | - | 用户未表达明显困难 |

**处理策略**：
- 不扣执行分
- 触发平台期解释（AI）
- 饮食方法微调候选
- 防焦虑流失

### 9.2 假平台期（False Plateau）

**判定条件**（满足任一）：

| 条件 | 阈值 | 说明 |
|------|------|------|
| 连续体重缺失 | ≥3天 | 无体重数据 |
| 执行分 | <60 | 任务执行不达标 |
| 能量分 | <50 | 用户能量状态差 |
| 隐性掉队信号 | ≥2次 | 不打卡、不回复等 |
| 阻碍主诉 | 明确表达 | 用户明确表达困难 |

**处理策略**：
- 按正常风险处理
- 执行分/能量分正常扣减
- 优先解决执行问题
- 必要时L2干预

### 9.3 平台期判定流程

```
每日体重数据检查
    ↓
有数据? → 是 → 波动<0.2kg? → 是 → 检查其他条件 → 全部满足 → 真平台期
    ↓                                    ↓
   否                              任一不满足 → 假平台期/正常波动
    ↓
无数据? → 是 → 缺失≥3天? → 是 → 假平台期
    ↓                    ↓
   否              否 → 正常待观察
```

## 12. 分数规则详解

### 12.1 执行分（execution_score）

**默认值**：60

#### 12.1.1 加分项

| 加分项 | 分值 | 触发条件 | 备注 |
|--------|------|----------|------|
| 当日主任务按时完成 | +2/项 | 当日21:00前完成 | 每日最多+6 |
| 连续3天完成今日三件事 | +5 | 连续3天完成率100% | - |
| 连续7天完成率>=80% | +8 | 连续7天完成率达标 | - |
| 主动补录有效数据 | +1 | 补录过往数据且有效 | 每日最多+2 |
| 有效回复主动问询 | +1 | AI判定为有效回复 | - |
| 完成当周复评 | +8 | 周复评完成 | - |
| **隐性加分：正常波动期不扣分** | 0 | 处于S3真平台期 | 平台期保护 |

#### 12.1.2 减分项

| 减分项 | 分值 | 触发条件 | 备注 |
|--------|------|----------|------|
| 单项任务逾期 | -2/项 | 超过截止时间完成 | - |
| 单项任务未完成 | -4/项 | 当日未完成 | - |
| 连续2天核心任务未完成 | -6 | 连续2天 | - |
| 连续3天核心任务未完成 | -12 | 连续3天 | 触发S4 |
| 追问已触发但无回复 | -2 | 24h内无回复 | - |
| 低质量打卡 | -2 | AI判定质量低 | - |
| 周复评触发后48h未开始 | -6 | 超时未开始 | - |
| **隐性减分：假平台期正常扣** | - | 处于S3假平台期 | 按正常规则 |

#### 12.1.3 使用原则

- 平台期（真平台期）不直接扣执行分
- 用户主动说明客观困难时，执行分扣减减半
- 投诉和医疗边界不走执行分路径

## 13. 调整动作类型（前置详述）

> **重要说明**：由于第14点之前已涉及具体调整动作策略，本节将调整动作类型前置并详细说明。

### 13.1 任务调整

| 动作 | 说明 | 适用场景 | 验证周期 |
|------|------|----------|----------|
| 缩小任务包 | 减少每日任务数量，从3件减到2件 | 用户连续未完成、能量低 | 3天 |
| 任务降级 | 将任务难度降低一档 | 任务执行困难、假平台期 | 3天 |
| 替代任务 | 用同类简单任务替代 | 任务明显不适配 | 3天 |
| 周末模式 | 切换到周末任务配置 | 周五18:00预生成触发 | 3天 |

### 13.2 行为难度调整

**定义**：在不改变目标和核心动作的前提下调整参数。

**可调整参数**：
- 次数（frequency）
- 频次（interval）
- 时长（duration）
- 执行窗口（time_window）
- 执行场景（context）

**示例**：
| 原任务 | 调整后 |
|--------|--------|
| 饭后站立10分钟 | 饭后站立3分钟 |
| 每天做3次深呼吸 | 每天做1次 |
| 睡前1小时关手机 | 先提前15分钟 |
| 每天喝8杯水 | 每天喝4杯水 |

**触发条件**：
- behavior_score 在 2.6 ~ 3.4 之间
- 用户反馈"有点难"
- timing_friction 信号触发

### 13.3 行为替换

**定义**：AI基于原任务目标生成一个更好执行的新任务，**不是**从行为库直接选一个旧任务。

**约束条件**：
1. **目标不变** - 保持原任务的核心健康目标
2. **成本更低** - 降低执行门槛
3. **更适配当前场景** - 匹配用户当前生活状态

**替换后处理**：
- 新任务进入 P1 顶部（最高优先级）
- 原任务标记为 `replaced`

**触发条件**：
| 条件 | 分值范围 | 动作 |
|------|----------|------|
| 行为评分较低 | 1.8 <= behavior_score < 2.6 | 优先行为替换 |
| 核心动作不合理但目标合理 | - | 行为替换 |
| 用户明确拒绝该方向 | - | 任务移除 |

**替换示例**：

| 原任务 | 替换原因 | 替换后任务 |
|--------|----------|------------|
| 每天100个开合跳 | 用户反馈膝盖疼 | 每天10分钟散步 |
| 睡前冥想15分钟 | 用户反馈没时间 | 午间深呼吸3次 |
| 记录每餐热量 | 用户反馈太复杂 | 只记录午餐热量 |

### 13.4 任务移除

**定义**：将当前行为任务从任务池中移除。

**可移除条件**（满足任一）：
- Day1 明确反馈"太简单/没必要/不适合"
- 连续不打卡且不回应（>=3天）
- 评分极低（behavior_score < 1.8）
- 用户明确拒绝继续

**移除前必须执行**：
1. 必须解释原因（AI话术）
2. 必须让用户确认（需要用户交互确认）

**移除后处理**：
- 次日切换下一既有任务
- 当前任务标记为 `removed`
- 标签保留在 confirmed 或 intervening

### 13.5 饮食方法微调

| 调整类型 | 说明 | 适用场景 |
|----------|------|----------|
| 热量微调 | ±50-100kcal | 体重下降过快/过慢 |
| 餐次调整 | 从3餐改2餐，或加餐 | 用户反馈饿/撑 |
| 食材替换 | 同类食材替换 | 用户反馈不爱吃/过敏 |
| 做法简化 | 复杂做法→简单做法 | 用户反馈没时间做 |

| 切换类型 | 说明 | 触发条件 |
|----------|------|----------|
| Plan B | 备用方案激活 | 主方案执行困难 |
| 阶段重评估 | 重新评估当前阶段 | 连续2周未达标 |

## 15. 全生命周期分阶段 PRD（表格版）

### 15.1 阶段A：首次测评

| 项目 | 内容 |
|------|------|
| **覆盖状态** | NOT_STARTED, IN_PROGRESS, INTERRUPTED, COMPLETED, EXPIRED, CANCELED |
| **监督重点** | 是否进入首次测评、是否中断、是否长时间停留不完成、是否出现非服务适配风险 |
| **显性监督** | 开始测评(assessment.start)、中断退出(assessment.interrupt)、完成测评(assessment.complete)、超时未完成、高风险答案命中 |
| **隐性监督** | 多次进入又退出(>=3次)、长时间停留不提交(>10min)、多次答非所问、连续极短回复(<5字) |
| **核心信号(中文)** | assessment_resistance(测评抵抗)、assessment_timeout(测评超时)、service_ineligible_risk(不可服务风险) |
| **核心动作** | 首次中断→首页恢复卡+AI轻召回；48小时内第二次中断→缩短入口说明，优先建议语音或简化测评；72小时仍未继续→降低主动催促频率；高风险或不可服务→终止当前测评并解释边界 |

### 15.2 阶段B：报告已出，待继续获取方案

| 项目 | 内容 |
|------|------|
| **监督重点** | 用户是否理解报告、是否愿意继续获取方案、是否被成本、难度、信任问题阻断 |
| **显性监督** | 报告打开(report.generate)、继续获取方案按钮点击、进入方案前阶段测评 |
| **隐性监督** | 多次看报告但不继续(>=2次，48h内)、高频问"有用吗/能瘦多少吗"、停留长但不点继续 |
| **核心信号(中文)** | plan_hesitation(方案犹豫)、trust_crisis(信任危机)、urgency_bias(求快心理) |
| **核心动作** | 优先解释报告，不先催转化；用AI解答"为什么需要继续获取方案"；必要时用首页卡或站内卡承接 |

### 15.3 阶段C：方案前阶段测评P1

| 项目 | 内容 |
|------|------|
| **监督重点** | 是否完成方案前补充信息、是否中断、是否因问题复杂产生疲劳 |
| **核心动作** | 中断后恢复入口；允许AI追问补齐；若多次中断，减少一次性问题量 |

### 15.4 阶段D：方案已出未激活或付费未支付

| 项目 | 内容 |
|------|------|
| **覆盖状态** | READY, AWAITING_PAYMENT |
| **监督重点** | 用户是否确认方案、用户是否激活免费方案、用户是否发起支付但未完成 |
| **显性监督** | 查看方案、方案确认、点击开始执行、点击支付、支付成功或失败 |
| **隐性监督** | 多次看方案不激活、反复问原理但不确认、发起支付后不完成(24h) |
| **核心信号(中文)** | plan_hesitation(方案犹豫)、payment_abandon(支付放弃) |
| **核心动作** | 免费方案→优先解释后促激活；付费方案→解释权益、流程和限制，不夸大承诺；支付发起未完成→站内卡优先，必要时Push，一日最多1次 |

### 15.5 阶段E：付费后待发货、待签收

| 项目 | 内容 |
|------|------|
| **覆盖状态** | AWAITING_DELIVERY |
| **监督重点** | 发货是否正常、物流是否异常、签收是否成功 |
| **核心规则** | 发货和签收以实时回调为主；对账轮询仅做兜底；签收后次日才进入付费Day1 |
| **核心动作** | 发货成功→即时Push；签收成功→即时Push+Day1预告；回调异常→优先对账，不先骚扰用户 |

### 15.6 阶段F：激活后待Day0

| 项目 | 内容 |
|------|------|
| **监督重点** | 用户是否完成Day0准备、是否具备执行条件 |
| **显性监督** | Day0页面打开、Day0任务完成 |
| **隐性监督** | 多次打开Day0页面不执行(day0.open_no_action)、只看不做 |
| **核心信号(中文)** | day0_stuck(Day0卡住) |
| **核心动作** | Day0只强调准备，不做强执行评价；Day0截止前做一次轻提醒；当天未做完，次日首页优先继续承接 |

### 15.7 阶段G：执行期Day1-Day6

| 项目 | 内容 |
|------|------|
| **监督重点** | 今日三件事是否执行、饮食、运动、行为三类任务是否闭环、是否出现掉队、低能量、外食、夜宵、失眠、出差等场景 |
| **核心规则** | 正常→L0或轻反馈；单次未完成→L1；连续未完成→L2；高风险→L3 |
| **信号覆盖** | 执行信号、行为任务信号、场景信号、交互信号 |

### 15.8 阶段H：Day7/14/21周复评

| 项目 | 内容 |
|------|------|
| **监督重点** | 用户是否点击复评入口、是否完成复评、目标进度是否达标、不达标是执行不足还是方法问题 |
| **核心规则** | 08:00出复评入口；当曰20:00未开始→Push；+24h未开始→再次提醒；+48h未完成→进入超时逻辑 |
| **复评后调整** | 可做：饮食方法微调、今日任务与下周任务参数微调、行为优先级调整、运动强度轻调；不可做：不切服务阶段、不重生成下一阶段方案 |
| **核心动作** | 复评完成后可微调；超时按流程处理 |

### 15.9 阶段I：Day28阶段复评

| 项目 | 内容 |
|------|------|
| **监督重点** | 用户是否进入阶段复评、是否达到当前阶段目标、未达标原因是什么 |
| **核心规则** | Day28 08:00出复评入口；当曰20:00未开始→Push；+24h未开始→Push+首页强提醒；+48h未完成→升级L2 |
| **达标处理** | 当前阶段完成→进入下一服务阶段P2/P3/...→生成下一阶段方案→免费方案确认后次日Day1→付费方案支付并签收后次日Day1 |
| **未达标处理** | 不直接进入下一阶段，必须先归因；归因至少落入以下类型之一 |
| **归因类型** | A1-执行不足、A2-饮食方法不匹配、A3-能量或生理波动、A4-场景干扰过强、A5-产品或履约问题 |
| **归因后下一步** | 1.先归因；2.判断是否属于服务内已充分交付；3.进入服务终局判定；4.若涉及退款，走退款评估流程 |

### 15.10 阶段J：终止与退款评估

| 项目 | 内容 |
|------|------|
| **覆盖状态** | TERMINATED |
| **终止原因** | Day28未达标后归因进入终局终止；提前终止规则命中；高风险或投诉导致服务终止 |
| **终止后仍需监督** | 退款流程进度、投诉闭环、人工跟进是否完成、用户是否点击重新开始 |
| **终止后停止** | 不再生成主动执行任务、不再发日常任务型Push、不再推进普通阶段升级 |

## 16. 行为干预任务 PRD

### 16.1 任务定位

行为干预是"今日三件事"中的第3个任务。任务来源于行为库，分两类：

| 任务类型 | 执行周期 | 目标 | 结束条件 |
|----------|----------|------|----------|
| 数据追踪类 | 1天 | 获取数据 | 完成后结束 |
| 行为习惯类 | 3天 | 形成习惯 | Day3后输出行为评分和动作 |

### 16.2 行为追问的四个评分维度

| 维度 | 权重 | 说明 |
|------|------|------|
| 真实性 | 30% | 用户是否真实执行 |
| 执行质量 | 25% | 执行动作是否规范 |
| 效果感知 | 20% | 用户是否感受到效果 |
| 可持续性 | 25% | 用户是否认为可以长期坚持 |

**建议公式**：
```
behavior_score = 0.30*真实性 + 0.25*执行质量 + 0.20*效果感知 + 0.25*可持续性
```

### 16.3 Day1/Day2/Day3目标

| 阶段 | 目标 |
|------|------|
| Day1 | 确认是否真实执行；判断是否明显不适配 |
| Day2 | 判断执行质量；判断是否开始产生效果；识别具体阻碍 |
| Day3 | 判断是否可形成习惯；形成最终行为评分 |

### 16.4 3天评估动作矩阵

| 完成情况 | 判定 | 动作 |
|----------|------|------|
| 0/3或1/3完成，且无回复 | 隐性掉队 | 当前任务退出执行位→标签保留→当前任务降入P3→次日切换行为库下一既有任务 |
| 0/3或1/3完成，但用户明确说"很难/没时间/不想做" | 核心动作仍合理 | 先难度调整 |
| | 核心动作不合理但目标合理 | 行为替换 |
| | 用户明确拒绝该方向 | 任务移除 |
| 2/3完成，但Day3失败 | - | 优先难度调整；多次同类阻碍再做替换 |
| 3天完成，behavior_score>=4.2 | 优秀 | 继续巩固或视为完成→流转下一任务→对应负向标签进入weakened |
| 3天完成，3.4<=behavior_score<4.2 | 一般 | 继续当前任务→不升级难度 |
| 3天完成，2.6<=behavior_score<3.4 | 较差 | 优先难度调整 |
| 3天完成，1.8<=behavior_score<2.6 | 差 | 优先行为替换→新任务进入P1顶部 |
| 3天完成，behavior_score<1.8 | 极差 | 当前任务移除→次日切换下一既有任务→标签保留 |

## 17. 标签流转规则

### 17.1 标签状态

| 状态 | 说明 |
|------|------|
| suspected | 怀疑/待确认 |
| confirmed | 已确认 |
| intervening | 干预中 |
| weakened | 弱化中 |
| resolved | 已解决 |

### 17.2 标签进入

| 进入条件 | 触发 |
|----------|------|
| 测评命中 | 测评直接命中 |
| 主动问询承认 | 用户主动承认 |
| AI语义识别明确主诉 | AI明确识别 |
| 隐性信号重复出现>=2 | 系统检测 |

### 17.3 标签升级

| 触发条件 |
|----------|
| 7天内同类问题重复出现 |
| 当前行为任务失败 |
| 隐性和显性反馈同时命中 |

### 17.4 标签弱化

满足任一即可：
- 连续1个行为周期 behavior_score >= 4.2 且 7天内无重复负向信号

### 17.5 标签解决

满足以下组合：
- 连续2个相关任务周期表现良好
- 或连续14天无相关显性和隐性信号
- 若有客观数据，不恶化

### 17.6 标签保留

以下情况下标签保留在 confirmed 或 intervening：
- 任务未完成
- 行为评分低
- 用户明确拒绝
- 隐性负向信号仍持续

## 18. 方案调整规则

### 18.1 周期内微调

28天服务周期内允许的主要调整：
- 饮食方法
- 饮食任务参数
- 运动任务难度、频次、时长
- 行为任务难度和顺序
- 提醒频率

**不直接调整**：
- 服务阶段本身不变
- 不因单次波动切换下一阶段
- 运动和行为优先做任务级调整，不做大阶段重生成

### 18.2 Day7/14/21周复评后的调整

**可做**：
- 饮食方法微调
- 任务参数微调
- 行为优先级调整
- 运动强度轻调

**不可做**：
- 服务阶段切换
- 下一阶段方案生成

### 18.3 Day28阶段复评后的调整

**达标**：
- 当前阶段完成
- 进入下一阶段 P2/P3/...
- 生成下一阶段方案

**未达标**：
- 未达标不直接进入下一阶段，必须先归因
- 归因类型：A1执行不足、A2饮食方法不匹配、A3能量或生理波动、A4场景干扰过强、A5产品或履约问题
- 处理流程：归因 → 服务终局判断 → 必要时退款评估

## 22. 验收标准

本模块的验收标准不是"页面做出来了"，而是以下闭环是否成立：

### 22.1 监督有效
- 系统能识别显性与隐性监督事件
- 体重、签收等关键事件实时优先
- 定时检测只做缺失和兜底

### 22.2 判定可解释
- 能说清楚某次动作为什么触发
- 能回放 event → signal → score → action

### 22.3 动作可执行
- 任务调整能落到实际任务
- 行为替换能生成新任务并进入P1
- 终止和退款评估有明确入口

### 22.4 渠道不打扰
- Push有冷却和优先级
