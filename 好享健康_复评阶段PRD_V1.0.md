## 一、文档目的

本文档定义好享健康 APP 中"复评阶段"的完整产品方案，明确复评触发条件、用户流程、监督规则、调整策略和验收标准。

复评阶段是监督与调整模块的核心环节，承担以下关键职责：

1. **周期性评估** - 在 Day7、Day14、Day21 和 Day28 对用户进展进行系统评估
2. **目标达成判定** - 判断用户是否达到当前阶段目标
3. **归因分析** - 对未达标情况进行归因分类
4. **策略调整** - 基于评估结果调整饮食、运动、行为方案
5. **阶段流转** - 决定是否进入下一服务阶段或终止服务

## 四、复评用户流程

### 4.1 周复评流程

```
用户点击"立即复评"
    ↓
进入周复评页面
    ↓
1. 查看本周完成情况（系统自动计算）
    - 任务完成率
    - 体重趋势
    - 行为任务完成情况
    ↓
2. 回答复评问题（5-8题）
    - 本周执行感受
    - 遇到的主要困难
    - 饮食/运动/行为反馈
    ↓
3. 确认提交
    ↓
4. 系统生成下周方案微调建议
    ↓
5. 用户确认调整
    ↓
6. 完成周复评
```

### 4.2 阶段复评流程

```
用户点击"立即复评"
    ↓
进入阶段复评页面
    ↓
1. 查看阶段目标完成情况
    - 阶段目标达成度（百分比）
    - 核心指标达成情况
    - 体重/围度变化趋势
    - 任务完成统计
    ↓
2. 回答阶段复评问题（10-15题）
    - 整体执行感受
    - 方案适配度评价
    - 困难与障碍分类
    - 是否愿意继续
    ↓
3. 系统判定目标达成状态
    ↓
    ├─ 达标（>=80%） → 进入下一阶段
    │   - 生成新阶段方案
    │   - 用户确认后次日Day1
    │
    └─ 未达标（<80%） → 进入归因流程
        - 收集归因信息
        - 判断服务内是否充分交付
        - 进入服务终局判定
```

## 五、复评监督规则

### 5.1 监督指标

| 指标类型 | 指标名称 | 计算方式 | 阈值 |
|----------|----------|----------|------|
| 完成率指标 | 周复评完成率 | 完成周复评用户数 / 应复评用户数 | >=85% |
| 时效性指标 | 复评及时完成率 | 20:00前完成数 / 总完成数 | >=70% |
| 质量指标 | 复评问题平均回答数 | 用户回答的问题数 / 总问题数 | >=80% |
| 达标指标 | 阶段目标达成率 | 达标用户数 / 完成阶段复评用户数 | >=60% |

### 5.2 异常监督信号

| 信号Code | 信号名称 | 触发条件 | 干预等级 | 处理动作 |
|----------|----------|----------|----------|----------|
| review_not_started | 复评未开始 | 当日20:00仍未开始 | L1 | Push提醒 |
| review_late_start | 复评延迟开始 | +24h仍未开始 | L1 | 再次Push + 首页强提醒 |
| review_timeout | 复评超时未完成 | +48h仍未完成 | L2 | 升级干预，可能L3 |
| review_abandon | 复评中途放弃 | 开始但未提交 | L1 | 次日轻提醒 |

### 5.3 复评期间优先规则

**重要**：复评进行期间，系统应优先完成复评，暂缓其他非紧急监督与调整动作。

| 优先级 | 处理规则 |
|--------|----------|
| P1 | 正在进行的复评流程 |
| P2 | 紧急风险事件（高危红线、医疗边界） |
| P3 | 日常任务打卡监督 |
| P4 | 日常问询 |
| P5 | 商业转化提醒 |

## 六、目标达成判定规则

### 6.1 阶段目标类型

| 目标类型 | 说明 | 权重 |
|----------|------|------|
| **体重目标** | 阶段减重目标达成 | 40% |
| **完成率目标** | 任务完成率达标 | 30% |
| **行为目标** | 行为任务完成情况 | 20% |
| **用户满意度** | 用户自评满意度 | 10% |

### 6.2 目标达成计算公式

```
目标达成度 = 
    体重达成分×0.4 + 
    完成率达成分×0.3 + 
    行为达成分×0.2 + 
    满意度分×0.1
```

**判定标准**：
- **达标**：目标达成度 >= 80%
- **基本达标**：70% <= 目标达成度 < 80%
- **未达标**：目标达成度 < 70%

### 6.3 各目标分项计算

#### 6.3.1 体重达成分计算

| 条件 | 得分 |
|------|------|
| 实际减重 >= 目标减重 × 1.2 | 100分 |
| 目标减重 × 1.0 <= 实际减重 < 目标减重 × 1.2 | 80分 |
| 目标减重 × 0.8 <= 实际减重 < 目标减重 × 1.0 | 60分 |
| 实际减重 < 目标减重 × 0.8 | 40分 |
| 体重反弹 | 0分 |

#### 6.3.2 完成率达成分计算

| 条件 | 得分 |
|------|------|
| 任务完成率 >= 90% | 100分 |
| 80% <= 完成率 < 90% | 80分 |
| 70% <= 完成率 < 80% | 60分 |
| 完成率 < 70% | 40分 |

#### 6.3.3 行为达成分计算

| 条件 | 得分 |
|------|------|
| 行为任务平均分 >= 4.0 | 100分 |
| 3.5 <= 平均分 < 4.0 | 80分 |
| 3.0 <= 平均分 < 3.5 | 60分 |
| 平均分 < 3.0 | 40分 |

#### 6.3.4 满意度分计算

| 用户自评 | 得分 |
|----------|------|
| 非常满意 | 100分 |
| 满意 | 80分 |
| 一般 | 60分 |
| 不满意 | 40分 |
| 非常不满意 | 0分 |

## 七、未达标归因分析

### 7.1 归因分类框架

当阶段复评未达标（目标达成度 < 70%）时，必须进行归因分析。归因分为五大类：

| 归因代码 | 归因名称 | 说明 | 触发条件 |
|----------|----------|------|----------|
| A1 | 执行不足 | 用户执行不充分 | 任务完成率<70%，执行分<45 |
| A2 | 饮食方法不匹配 | 当前饮食方案不适合用户 | 用户反馈饮食难执行/热量控制困难 |
| A3 | 能量或生理波动 | 用户能量状态低或生理期影响 | 能量分<50 或 生理期记录 |
| A4 | 场景干扰过强 | 外部场景（出差/加班/外食）频繁干扰 | 场景信号 >=3次 |
| A5 | 产品或履约问题 | 产品本身问题或商品履约问题 | 用户反馈产品问题或物流问题 |

### 7.2 归因判定流程

```
用户提交阶段复评 → 目标达成度<70%?
       ↓ 是
触发归因分析
    ↓
1. 自动系统归因（基于分数和信号）
    - 执行分<45 → A1
    - 能量分<50 → A3
    - 场景信号>=3 → A4
    - 体重数据缺失率>30% → A1
    ↓
2. 用户主动归因（复评问题答案分析）
    - AI语义分析用户反馈
    - 识别用户自述原因
    ↓
3. 综合判定
    - 系统归因 + 用户归因 = 最终归因
    - 可多选（最多2个）
```

### 7.3 归因后处理路径

```
未达标 + 归因 → 判断服务内是否充分交付 → 是 → 进入服务终局判定
                                              ↓
                                          否 → 方案调整 → 延长服务期
```

### 7.4 各归因类型的处理策略

| 归因类型 | 核心问题 | 处理策略 | 后续动作 |
|----------|----------|----------|----------|
| A1 执行不足 | 用户没做或做得少 | 1. 缩小任务包<br>2. 降低难度<br>3. 加强提醒 | 若调整后仍不执行 → 进入S4风险期 → 考虑终止 |
| A2 饮食方法不匹配 | 方案不适合用户 | 1. 饮食方法微调<br>2. 食材替换<br>3. 简化做法 | 最多2次调整 → 仍不适应 → 考虑换方案或终止 |
| A3 能量或生理波动 | 用户身体状态问题 | 1. 优先恢复能量<br>2. 降低运动强度<br>3. 延长期望值 | 能量恢复后重新评估 |
| A4 场景干扰过强 | 外部环境不可控 | 1. 场景化方案（出差模式/周末模式）<br>2. 提高场景适应性 | 若场景持续 → 考虑用户不匹配服务 |
| A5 产品或履约问题 | 我方交付问题 | 1. 产品问题 → 优化或更换<br>2. 履约问题 → 补偿或加急 | 属于我方责任 → 考虑退款或延长服务 |

## 八、复评后方案调整规则

### 8.1 调整原则

| 原则 | 说明 |
|------|------|
| 周复评可微调，阶段复评可重定 | 周复评只做微调，阶段复评可做重大调整 |
| 不切服务阶段 | 周复评后不切换服务阶段（S1→S2等） |
| 不重生成下一阶段方案 | 周复评不生成新阶段方案，只调参数 |
| 调整需用户确认 | 所有调整必须用户确认后才生效 |
| 记录调整历史 | 每次调整需记录原因和预期效果 |

### 8.2 周复评后可调整内容

**可调整**：

| 调整类型 | 具体内容 | 调整幅度 |
|----------|----------|----------|
| 饮食方法微调 | 热量±50-100kcal/日 | 小幅度 |
| 任务参数微调 | 运动时长±10%、频次调整 | 小幅度 |
| 行为优先级调整 | 调整行为任务顺序 | 优先级变化 |
| 运动强度轻调 | 强度档位微调 | 一档以内 |

**不可调整**：

| 类型 | 说明 |
|------|------|
| 服务阶段切换 | 不从S1切到S2等 |
| 下一阶段方案生成 | 不生成新的阶段方案 |
| 大目标调整 | 不减重目标大幅修改 |

### 8.3 阶段复评后可调整内容

#### 8.3.1 达标后的调整

**达标条件**：目标达成度 >= 80%

**处理流程**：

```
阶段达标 → 当前阶段完成 → 进入下一服务阶段（P2/P3/...）→ 生成下一阶段方案
    ↓
免费方案：用户确认后次日Day1
付费方案：支付并签收后次日Day1
```

**可调整**：

| 类型 | 说明 |
|------|------|
| 阶段目标调整 | 下一阶段目标适度提高（+5-10%） |
| 方案升级 | 饮食方案更严格，运动强度提升 |
| 行为任务进阶 | 升级到更高级行为习惯 |
| 提醒频率下调 | 进入稳态期后降低打扰 |

#### 8.3.2 未达标后的调整

**未达标条件**：目标达成度 < 70%

**处理流程**：

```
未达标 → 归因分析（A1-A5）→ 判断服务内是否充分交付 → 是 → 进入服务终局判定
                                                      ↓
                                                  否 → 方案调整 → 延长服务期
```

**调整策略**（根据归因）：

| 归因 | 调整动作 |
|------|----------|
| A1 执行不足 | 1. 缩小任务包（3件→2件）<br>2. 降低难度<br>3. 加强监督和提醒<br>4. 若调整后仍不执行 → 进入S4风险期 → 考虑终止 |
| A2 饮食方法不匹配 | 1. 更换饮食方法（如地中海→低碳）<br>2. 简化操作流程<br>3. 提供更多食谱选项<br>4. 最多调整2次，仍不适应 → 换方案或终止 |
| A3 能量或生理波动 | 1. 优先恢复能量（降低强度）<br>2. 延长期望值<br>3. 生理期后重新评估<br>4. 能量恢复后重新制定目标 |
| A4 场景干扰过强 | 1. 提供场景化方案（出差模式）<br>2. 提高任务场景适应性<br>3. 增加灵活性<br>4. 若场景持续不可控 → 考虑用户不匹配服务 |
| A5 产品或履约问题 | 1. 产品问题 → 优化或更换<br>2. 履约问题 → 补偿或加急<br>3. 属于我方责任 → 考虑退款或延长服务 |

## 九、复评期间用户反馈处理

### 9.1 反馈收集机制

复评阶段必须收集以下用户反馈：

| 反馈类别 | 问题示例 | 收集方式 |
|----------|----------|----------|
| **执行感受** | "本周执行难度如何？" | 评分题（1-5分） |
| **困难归因** | "遇到的主要困难是什么？" | 多选题+开放题 |
| **方案适配** | "饮食方案适合你吗？" | 评分题 |
| **身体感受** | "运动后感觉如何？" | 选择题（轻松/刚好/累/极限） |
| **继续意愿** | "是否愿意继续？" | 选择题（是/否/需要调整） |
| **开放反馈** | "其他想说的？" | 开放题 |

### 9.2 反馈优先级

| 反馈类型 | 处理优先级 | 响应时限 |
|----------|------------|----------|
| 负向情绪（想放弃、沮丧） | P0-即时 | 立即响应 |
| 求助 | P0-即时 | 立即响应 |
| 方案调整请求 | P0-即时 | <1h |
| 任务调整请求 | P1-高 | <2h |
| 一般建议 | P2-中 | <24h |
| 正向反馈 | P3-低 | 记录即可 |

### 9.3 反馈处理策略

#### 9.3.1 负向情绪即时响应

**识别信号**：
- 用户选择"非常困难"或"不想继续"
- AI语义识别到沮丧、焦虑、失望等情绪
- 用户明确表达"想放弃"

**响应流程**：
```
识别负向情绪 → AI即时共情 → 询问具体原因 → 根据原因采取动作
    ↓
若能量低 → 立即缩小任务包 + 降低难度
若方案不适 → 记录原因，进入方案调整流程
若场景问题 → 提供场景化方案
若情绪问题 → AI安慰 + 必要时人工介入
```

#### 9.3.2 方案调整请求处理

**识别信号**：
- 用户明确说"想换方案"
- 用户说"目标完不成"
- 用户说"饮食/运动不适合"

**响应流程**：
```
识别调整请求 → AI确认具体需求 → 收集调整原因 → 评估可行性 → 生成调整建议 → 用户确认 → 执行调整
```

**调整可行性判断**：

| 调整类型 | 可调整次数 | 条件 |
|----------|------------|------|
| 目标调整 | 1次/28天 | 需复评后判定 |
| 方案微调 | 2次/28天 | 有明确不适配证据 |
| 方案切换 | 1次/阶段 | 当前方案确实不匹配 |
| Plan B激活 | 不限 | 紧急情况 |

#### 9.3.3 任务调整请求处理

**识别信号**：
- 用户说"任务太多了"
- 用户说"这个做不来"
- 用户说"想换个任务"

**响应流程**：
```
识别调整请求 → AI确认具体任务 → 判断调整类型 → 执行调整
    ↓
难度调整：参数调整（次数/频次/时长）
行为替换：AI生成新任务进入P1
任务移除：解释原因并确认
```

## 十一、复评后方案调整示例

### 11.1 周复评调整示例

**场景**：用户Day7周复评，完成率75%，用户反馈"这周工作太忙，没时间做饭"

**系统判定**：
- 完成率：75% → 得分60（中等）
- 用户反馈：场景干扰（加班场景3次）
- 能量分：65（正常）

**调整建议**：
1. 下周任务包缩小：3件→2件
2. 提供加班场景饮食方案（外卖选择指南）
3. 运动调整为周末集中模式
4. 提醒时间延后到20:00

**用户确认** → 下周方案更新生效

### 11.2 阶段复评达标示例

**场景**：用户Day28阶段复评，目标达成度85%

**达成情况**：
- 体重目标：达成92%
- 完成率：88%
- 行为任务平均分：4.1
- 满意度：满意（4分）

**处理**：
1. 判定：达标
2. 进入下一阶段（S3平台期风险期）
3. 生成新阶段方案：
   - 目标调整：减重目标提高5%
   - 饮食方法：维持当前，微调热量
   - 运动强度：提升一档
   - 行为任务：升级到习惯级
4. 用户确认后，次日Day1

### 11.3 阶段复评未达标归因示例

**场景**：用户Day28阶段复评，目标达成度62%

**归因分析**：
- 系统归因：执行分42（低）、场景信号4次（外食/出差）
- 用户归因：用户反馈"总在外面吃，方案执行不了"、"运动时间不够"
- 综合归因：A1（执行不足） + A4（场景干扰）

**判定**：未达标

**处理流程**：
1. 判断服务内是否充分交付 → 是（方案已提供，但用户执行困难）
2. 进入服务终局判定
3. 用户选择：继续服务（方案调整）或终止

**方案调整**（若用户选择继续）：
1. 缩小任务包：3件→2件
2. 提供外食场景饮食指南
3. 运动调整为出差友好模式（无需器械）
4. 行为任务替换为更易场景适配的任务
5. 延长服务期7天，重新评估
