# 饮食打卡

### 1. 需求概述

该模块是用户全天饮食计划的指令中枢，将复杂的饮食方案拆解为可操作的原子化任务。系统通过“计划态”与“记录态”的动态流转，确保用户不仅知道“吃什么”，还能实时反馈“吃了什么”，从而驱动AI智能体进行下一环节的监测与调整。

### 2. 页面模块总览

| 区域 | 模块名称 | 核心功能 |
| --- | --- | --- |
| 顶部 | **AI管家动态话术** | 根据时间、用户状态、执行进度生成动态话术。 |
| 中部 | **今日能量代谢看板** | 实时展示热量数据，包括余额计算、超标预警、营养素进度 |
| 下部 | **今日食材清单** | 按餐次展示具体食材及可替换选项。 |
| 底部 | **操作按钮区** | 提供打卡、补录、拍照 / 语音录入等核心操作入口 |

### 3. 核心概念与状态流转

#### 3.1 核心概念定义

| 概念 | 定义 |
| --- | --- |
| **计划态** | 系统根据用户测评数据、减重目标等生成的未来饮食指令，包括具体餐次、食材、重量及可替换选项 |
| **记录态** | 用户通过拍照、语音或文字录入的实际摄入行为，系统识别后更新为“已记录”状态 |
| **动态流转** | 用户行为（打卡、补录、超标）触发系统状态变化，AI话术、数据看板、食材清单随之更新 |

#### 3.2 状态流转机制

| 触发行为 | 流转过程 | 系统响应 |
| --- | --- | --- |
| **打卡成功** | 计划态 → 记录态 | AI确认，数据看板刷新 |
| **摄入偏差** | 记录态（标记“偏差”） | AI话术切换为救助/止损建议 |
| **未打卡超时** | 计划态 → 补录态 | AI话术切换为补录引导 |

### 4. 功能模块详述

#### 4.1 AI管家动态话术

**功能描述**：根据用户当前健康状态（S1-S5）、执行进度及当前时间窗生成动态话术。

**话术结构**：共情句 + 判断句 + 动作指令 + 鼓励句

| 话术场景 | 触发条件 | 示例 |
| --- | --- | --- |
| **指令态** | 未打卡前 | “午餐时间到啦！🍱 别忘了按照方案吃这100g酱牛肉，记得拍个照让我看看哦。” |
| **确认/激励态** | 打卡后 | “午餐打卡很完美！✅ 刚才的蛋白和纤维配比非常标准，下午你会感觉精力充沛的。” |
| **救助/止损态** | 摄入偏差（如识别出高热量） | “这顿火锅看起来很过瘾，但脂肪确实高了些。🍲 没关系，今晚咱们清淡点补回来。” |
| **补录引导态** | 超过时间窗未打卡 | “刚才的午餐还没记呢，简单告诉我吃了什么，我来帮你补录，别让记录断档哦。” |

#### 4.2 今日能量代谢看板

1.  **功能描述**：实时展示今日能量代谢闭环数据，作为用户执行的“证据中心”。
    
2.  **核心数据看板**
    
    | 数据项 | 说明 | 计算规则/备注 |
    | --- | --- | --- |
    | **推荐总摄入** | 每日推荐摄入量 | 由方案Agent根据测评数据、减重方式自动生成，详阅【[饮食方案计算逻辑](http://axure.sys.hxsapp.net/KHN5C5/#id=a2pc54&p=%E9%A5%AE%E9%A3%9F%E6%96%B9%E6%A1%88%E8%AE%A1%E7%AE%97%E8%A7%84%E5%88%99%E3%80%900306%E3%80%91&g=1)】步骤一的“计算总能量和推荐摄入量”规则说明。 |
    | **饮食已摄入** | 当日已录入的所有餐次热量汇总 | 用户录入后1秒内刷新 |
    | **运动消耗** | 由运动任务打卡驱动 | 计算公式：MET值 × 体重(kg) × 运动时长(小时) |
    | **剩余可摄入** | 动态余额 | 计算公式：推荐总摄入 - 饮食已摄入 + 运动消耗 × 0.9<br>超标警示：数值变红/负数，进度条视觉预警 |
    
3.  **三大营养素：**蛋白质、脂肪、碳水化合物。
    

*   **需求概述**：为了确保摄入的“质量”而不仅是“数量”，系统对三大宏量营养素进行原子化监测。
    
*   **展示形式**：每个营养素配有双色进度条，实现推荐 vs. 实际的直观对比。
    
*   **超标规则**：当某项指标（如脂肪）摄入超过计划额度时，该营养素进度条自动切换为警示色。（并触发AI管家推送纠偏建议 - 待定）
    

#### 4.3 今日食材清单

1.  功能描述：按用户餐次（如：早、午、加、晚）展示具体食材（含重量）、及可替换食材。
    
2.  推荐规则：详阅【[饮食方案计算逻辑](http://axure.sys.hxsapp.net/KHN5C5/#id=a2pc54&p=%E9%A5%AE%E9%A3%9F%E6%96%B9%E6%A1%88%E8%AE%A1%E7%AE%97%E8%A7%84%E5%88%99%E3%80%900306%E3%80%91&g=1)】步骤七的“选择具体食材及替代食材”
    

| 字段 | 说明 |
| --- | --- |
| 餐次 | 通过【[饮食方案计算逻辑](http://axure.sys.hxsapp.net/KHN5C5/#id=a2pc54&p=%E9%A5%AE%E9%A3%9F%E6%96%B9%E6%A1%88%E8%AE%A1%E7%AE%97%E8%A7%84%E5%88%99%E3%80%900306%E3%80%91&g=1)】步骤四的“确定餐次数量及类型”确认用户每日餐次<br>    ○ 默认：4餐（早、中、下午、晚）<br>    ○ 疾病（血糖异常/胃肠差/易饥饿）：5餐（早、上午、中、下午、晚）<br>    ○ 夜班/通宵班：3餐（睡醒餐、上班前餐、宵夜） |
| 核心食材 | 详阅【[饮食方案计算逻辑](http://axure.sys.hxsapp.net/KHN5C5/#id=a2pc54&p=%E9%A5%AE%E9%A3%9F%E6%96%B9%E6%A1%88%E8%AE%A1%E7%AE%97%E8%A7%84%E5%88%99%E3%80%900306%E3%80%91&g=1)】步骤七的“选择具体食材及替代食材”说明 |
| 重量 | 根据BRM【[食物库](http://axure.sys.hxsapp.net/KHN5C5/#id=0ah6t1&p=%E9%A3%9F%E7%89%A9%E5%BA%93%E3%80%900306%E3%80%91&g=1)】的换算规则，转换为生活化描述，帮助用户直观理解食用量。比如：“1碗米饭”“1根玉米”“拳头大小”等 |
| 可替代食材 | *   推荐逻辑：推荐相同二级分类的食材，每项核心食材提供2-3种替代<br>    <br>*   重量展示：同三级分类不展示重量，否则需展示<br>    <br>*   排序优先级：同三级分类 > 同二级分类<br>    <br>*   **换一换：**点击后生成2-3种可替代食材，优先推荐不重复选项 |

#### 4.4 操作按钮区

1.  **功能描述**：底部按餐次展示打卡入口，用户点击对应餐次后弹出打卡方式浮层，选择拍照或文字/语音方式记录饮食。
    
2.  **餐次状态与交互**
    
    | 状态 | 定义 | 交互行为 |
    | --- | --- | --- |
    | **已完成** | 已成功发送图片或文字记录到对话流 | 可点击，再次记录同一餐次（支持多次补充） |
    | **进行中** | 当前时间在系统展示窗口内 | 可点击，进入正常打卡流程 |
    | **未开始** | 未到达系统展示时间窗口 | 点击无响应或提示“尚未到用餐时间” |
    | **未完成** | 超过推荐用餐时间未打卡 | 可点击，进入补录流程 |
    
3.  **打卡方式浮层**
    

用户点击餐次后，底部弹出浮层，提供两种打卡方式：

| 选项 | 点击后行为 |
| --- | --- |
| **拍照打卡** | 进入【拍照记录】页面，拍摄食物照片后发送至对话流 |
| **文字/语音打卡** | 打开【文字/语音记录】弹层，输入文字或语音转文字后发送至对话流 |

**说明**：同一餐次支持多次记录，用户可在“已完成”状态下继续补充，系统自动合并更新该餐次摄入数据。
