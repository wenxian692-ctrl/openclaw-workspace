# 运动打卡反馈收集

### 1. 需求概述

用户完成运动打卡后，系统需收集本次运动的主观感受反馈，用于评估运动强度是否合适、是否存在过度训练风险，并为后续运动方案动态调整提供依据。

根据打卡方式不同，反馈收集分为两种场景：

*   **逐项打卡**：在运动打卡弹层内直接收集单个运动的反馈
    
*   **一键打卡**：完成所有运动后，在对话流中由AI发起反馈收集
    

### 2. 触发条件

| 打卡方式 | 触发时机 | 收集方式 |
| --- | --- | --- |
| **逐项打卡** | 用户点击任一运动项目的【打卡】按钮后 | 在运动打卡弹层内弹出反馈选项 |
| **一键打卡** | 用户确认一键打卡后，所有运动标记为已完成 | 系统代发消息至对话流，AI发起反馈收集 |

### 3. 反馈收集内容

#### 3.1 核心反馈项

| 反馈项 | 提问意图 | 选项（单选） |
| --- | --- | --- |
| **运动强度感知** | 了解本次运动的吃力程度，判断强度是否合适 | • 轻松<br>• 刚好<br>• 挺累<br>• 极限了 |

#### 3.2 追问场景（仅一键打卡）

| 条件 | 处理方式 |
| --- | --- |
| **今日运动项目 > 1**<br>且<br>用户选择**“挺累”**或**“极限了”** | AI追问具体是哪个运动项目导致吃力（以便后续精准调整），需展示今日所有运动项目作为选项（可多选） |
| 其他情况 | 不触发追问，AI给予鼓励后结束反馈 |

### 4. 交互流程

#### 4.1 逐项打卡反馈（弹层内）

**触发**：用户点击运动项目右侧【打卡】按钮，该项目标记为“已完成”后，弹层内立即展示反馈选项。

**界面布局**：

*   在已完成项目的下方或附近位置，展示反馈问题“运动完感觉怎么样呢？”
    
*   下方排列四个选项：【轻松】【刚好】【挺累】【极限了】（单选）
    
*   用户选择任一选项后，反馈选项区域消失，无需额外确认
    

**数据记录**：

*   记录该运动项目的完成时间、用户反馈选项
    
*   不触发AI对话回复，反馈数据直接入库
    

#### 4.2 一键打卡反馈（对话流）

**触发**：

1.  用户点击顶部【一键打卡】并确认
    
2.  所有运动标记为“已完成”
    
3.  弹层自动关闭
    
4.  系统代发消息“我完成今日全部运动了”至对话流
    

### 5. 状态与数据同步

| 场景 | 同步规则 |
| --- | --- |
| 逐项打卡反馈 | 反馈数据实时入库，不影响今日任务卡状态 |
| 一键打卡反馈 | 对话流反馈完成后，数据入库；<br>若用户未完成反馈（如中途退出），标记为“待补充” \- 运动没反馈，还要再问吗？待定 |

### 6. 异常处理

| 场景 | 处理方式 |
| --- | --- |
| 用户逐项打卡后未选择反馈（直接关闭弹层） | 不强制收集，该运动标记为“已完成但无反馈” |
| 一键打卡后用户不回应反馈 | AI等待2分钟后结束会话，标记为“待补充”<br>\-待定 |
| 用户选择“挺累/极限了”后不选择具体动作 | 等待30秒后AI提示：“如果暂时想不起来，可以先这样，明天我会帮你留意强度”<br>\-待定 |

### 7. 话术语气规范

*   保持亲切、自然的语气，像朋友聊天
    
*   避免机械感、命令式或系统提示语，可根据用户回答灵活调整回应
    
*   差异化回应，给出有温度、有针对性的回应，避免千篇一律
    
*   适当使用表情符号增强亲和力（如🍽️、😋、🤔）
