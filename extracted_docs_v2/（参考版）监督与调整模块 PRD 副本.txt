监督与调整模块 PRD
1. 文档目的
本文档定义好享健康 APP 中“监督与调整”模块的完整产品方案，供产品、前端、后端、AI、数据、测试、顾问团队评审与落地。
本模块不是单纯的提醒系统，而是 AI Agent 驱动健康管理的控制层，负责在全生命周期内持续感知用户状态，识别信号，生成动作，并验证动作是否生效。
本模块覆盖：
首次测评
报告后继续获取方案
方案生成与待激活
付费待支付、待发货、待签收
Day0 与 Day1
28 天服务执行期
Day7/14/21 周复评
Day28 阶段复评
下阶段承接
服务终止与退款评估
2. 模块定义
监督与调整模块的核心职责是：
持续监督
信号识别
分数更新
生命周期切换
调整策略生成
触达与执行
结果回写
统一主链路如下：
监督来源 -> 事件 -> 信号 -> 分数 -> 生命周期 -> 干预等级 -> 调整动作 -> 回写 -> 再观察
3. 模块目标
3.1 用户目标
用户始终知道当前最重要的下一步
用户在困难时能被及时理解和拉回
用户不会因平台期、低能量、场景波动被误判
用户在高风险时能被及时保护和升级人工
3.2 业务目标
提升测评完成率
提升方案激活率
提升 Day0 与 Day1 完成率
提升今日三件事执行率
提升周复评与阶段复评完成率
降低掉队率、投诉率、无反馈率
3.3 系统目标
可解释
可追溯
可配置
可灰度
可审计
4. 核心设计原则
4.1 全生命周期持续监督
监督不是只发生在执行期，而是贯穿以下所有阶段：
测评前和测评中
方案待获取与待激活
履约等待期
Day0、Day1
执行期
复评期
阶段切换期
终止和退款评估期
4.2 显性与隐性并行
监督必须同时覆盖两类信息：
显性监督：用户明确完成、明确反馈、明确表达
隐性监督：不打卡、不回复、临近截止才完成、打开不执行、短回复、重复同一困难
4.3 规则、AI、混合判定分层
不是所有事情都让 AI 判定，也不是所有事情都适合死规则。
规则引擎负责明确阈值、明确时点、必须稳定的判断
AI 负责自然语言理解、阻碍识别、语义归因
混合判定负责 AI 结构化提取后由规则引擎落动作
4.4 风险优先级最高
任何阶段都遵循以下优先级：
医疗和安全边界
正在进行中的测评和复评
当前主任务闭环
风险纠偏
日常主动问询
商业转化和续费
4.5 调整必须回写和验证
系统不能只“调一次”，必须记录：
为什么调
调了什么
用户是否接受
调整后 3 天或 7 天是否改善
是否继续、恢复、升级或终止

正在复评时不能先发日常问询
待签收时不能提前进入付费 Day1

9.3 实时事件与定时检测
原则
实时事件是主链路
定时检测是缺失检测、超时检测、兜底对账、预定节点触发
必须保留的定时检测
每日 00:10 分数结算
每日 00:20 今日任务生成
每日 08:00 连续未打卡检测
Day7/14/21 复评入口触发
Day28 阶段复评入口触发
周五 18:00 周末任务预生成
方案待确认超时检测
复评未完成超时检测
Push 未响应后的升级检测
关于体重数据
体重采集本身不是定时产生，而是：
用户主动记录
体脂秤实时同步
定时检测保留的目的只有两个：
检查预期应有但未出现的体重数据
检查同步失败或长期无数据
建议规则：
当日 10:00 检查是否已有体重数据
若无，触发轻提醒候选
当日 20:00 再检查一次
连续 3 天无体重数据，形成 
weight_missing
关于商品签收
签收状态以物流回调为主，不以轮询作为主逻辑。
建议规则：
主链路：物流 API 回调实时更新签收
兜底链路：定时对账防回调丢失
建议对账：
发货后 2 小时无物流更新，做一次对账
派送中后每 4 小时对账一次
24 小时无回调且状态不明，生成履约异常事件

14. 行为干预任务 PRD
14.1 任务定位
行为干预是“今日三件事”中的第 3 个任务。
任务来源于行为库，分两类：
数据追踪类
执行 1 天
目标是获取数据
完成后结束
行为习惯类
以 3 天为一个任务周期
目标是形成习惯
每天打卡后追问
Day3 后输出行为评分和动作
14.2 行为追问的四个评分维度
真实性
执行质量
效果感知
可持续性
每个维度 1 到 5 分。
建议公式：
behavior_score = 0.30*真实性 + 0.25*执行质量 + 0.20*效果感知 + 0.25*可持续性
14.3 Day1/Day2/Day3 目标
Day1
确认是否真实执行
判断是否明显不适配
Day2
判断执行质量
判断是否开始产生效果
识别具体阻碍
Day3
判断是否可形成习惯
形成最终行为评分
14.4 行为任务可选动作
行为习惯类任务最终只有 5 类动作：
继续当前任务
难度调整
行为替换
任务移除
完成并流转到下一任务

14.6 难度调整规则
难度调整是在不改变目标和核心动作的前提下调参数：
次数
频次
时长
执行窗口
执行场景
示例：
“饭后站立 10 分钟” -> “饭后站立 3 分钟”
“每天做 3 次深呼吸” -> “每天做 1 次”
“睡前 1 小时关手机” -> “先提前 15 分钟”
14.7 行为替换规则
行为替换不是从行为库直接选一个旧任务，而是 AI 基于原任务目标生成一个更好执行的新任务。
约束：
目标不变
成本更低
更适配当前场景
替换后：
新任务进入 
P1
 顶部
原任务记为 
replaced

以下情况可移除：
Day1 明确反馈“太简单/没必要/不适合”
连续不打卡且不回应
评分极低
用户明确拒绝继续
移除前：
必须解释原因
必须让用户确认
移除后：
次日切换下一既有任务
当前任务记为

15. 隐性监督识别与策略输出
15.1 隐性信号识别
implicit_resistance
含义：
用户没有明确说拒绝
但持续回避任务
触发建议：
3 天内未打卡 >= 2
或打开不执行 >= 2
或不回追问 >= 2
首选动作：
缩小任务包
次选动作：
任务移除
fake_completion
含义：
点了打卡
但真实性可疑
触发建议：
打卡后连续极短回复，无法说明细节
或 AI 判定回答与任务动作不一致
首选动作：
增加一次核实追问
次选动作：
下调真实性分与执行质量分
timing_friction
含义：
不是不愿意做，是时间窗口不合适
触发建议：
连续 2 次以上临近截止才完成
首选动作：
改时间窗
降频次
context_mismatch
含义：
当前任务与当前生活场景不匹配
触发建议：
场景打断 >= 2
相同 barrier 连续出现 >= 2
首选动作：
行为替换
low_effect_signal
含义：
执行了，但没有感受到效果
触发建议：
effect_score <= 2
且 
real_done_score >= 3
首选动作：
行为替换
habit_not_forming
含义：
能做，但做不成习惯
触发建议：
Day3 后 
sustainability_score <= 2
且 
real_done_score >= 3
首选动作：
难度调整
