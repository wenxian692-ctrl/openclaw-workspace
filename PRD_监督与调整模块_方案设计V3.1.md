## 1. 文档目的

本文档定义好享健康 APP 中"监督与调整"模块的完整产品方案，供产品、前端、后端、AI、数据、测试、顾问团队评审与落地。

本模块不是单纯的提醒系统，而是 AI Agent 驱动健康管理的控制层，负责在全生命周期内持续感知用户状态，识别信号，生成动作，并验证动作是否生效。

**本模块覆盖**：
- 首次测评
- 报告后继续获取方案
- 方案生成与待激活
- 付费待支付、待发货、待签收
- Day0 与 Day1
- 28 天服务执行期
- Day7/14/21 周复评
- Day28 阶段复评
- 下阶段承接
- 服务终止与退款评估

## 2. 模块定义

监督与调整模块的核心职责是：

1. **持续监督** - 贯穿用户全生命周期
2. **信号识别** - 从事件中提取有效信号
3. **分数更新** - 维护三大核心分数
4. **生命周期切换** - 智能判断用户所处阶段
5. **调整策略生成** - 基于信号和分数生成动作
6. **触达与执行** - 选择合适渠道触达用户
7. **结果回写** - 记录并验证调整效果
8. **用户反馈收集** - 任何时点用户主动反馈的收集和处理

**统一主链路**：

```
监督来源 → 事件 → 信号 → 分数 → 生命周期 → 干预等级 → 调整动作 → 触达 → 执行 → 回写 → 再观察
```

## 3. 模块目标

### 3.1 用户目标

- 用户始终知道当前最重要的下一步
- 用户在困难时能被及时理解和拉回
- 用户不会因平台期、低能量、场景波动被误判
- 用户在高风险时能被及时保护和升级人工
- **用户的任何主动反馈都能被及时响应和处理**

### 3.2 业务目标

- 提升测评完成率
- 提升方案激活率
- 提升 Day0 与 Day1 完成率
- 提升今日三件事执行率
- 提升周复评与阶段复评完成率
- 降低掉队率、投诉率、无反馈率

### 3.3 系统目标

- 可解释
- 可追溯
- 可配置
- 可灰度
- 可审计

## 4. 核心设计原则

### 4.1 全生命周期持续监督

监督不是只发生在执行期，而是贯穿以下所有阶段：
- 测评前和测评中
- 方案待获取与待激活
- 履约等待期
- Day0、Day1
- 执行期
- 复评期
- 阶段切换期
- 终止和退款评估期

### 4.2 显性与隐性并行

监督必须同时覆盖两类信息：
- **显性监督**：用户明确完成、明确反馈、明确表达
- **隐性监督**：不打卡、不回复、临近截止才完成、打开不执行、短回复、重复同一困难

### 4.3 规则、AI、混合判定分层

- **规则引擎**：负责明确阈值、明确时点、必须稳定的判断
- **AI**：负责自然语言理解、阻碍识别、语义归因
- **混合判定**：负责 AI 结构化提取后由规则引擎落动作

### 4.4 风险优先级最高

任何阶段都遵循以下优先级：

1. 医疗和安全边界
2. **用户主动反馈（负向情绪、调整需求）**
3. 正在进行中的测评和复评
4. 当前主任务闭环
5. 风险纠偏
6. 日常主动问询
7. 商业转化和续费

### 4.5 调整必须回写和验证

系统不能只"调一次"，必须记录：
- 为什么调
- 调了什么
- 用户是否接受
- 调整后 3 天或 7 天是否改善
- 是否继续、恢复、升级或终止

## 5. 决策思维链

监督与调整模块的决策必须按以下8步执行：

### Step 1：识别当前所处阶段

先判断用户当前在哪个阶段：
- 首次测评
- 报告后待继续
- 方案待激活
- 付费待支付
- 付费待签收
- Day0
- 执行期
- 周复评
- 阶段复评
- 下阶段承接
- 终止

**注意**：系统不能跨阶段误触发。例如：
- 正在复评时不能先发日常问询
- 待签收时不能提前进入付费 Day1

### Step 2：收集监督事件

从显性事件和隐性事件中收集当前有效监督输入。

### Step 3：生成信号

对事件做归并和解释，得到策略可用的信号，而不是直接根据单一事件下动作。

### Step 4：更新分数

更新：
- 执行分（execution_score）
- 能量分（energy_score）
- 风险分（risk_score）

### Step 5：判断生命周期

根据分数、趋势、目标达成情况更新：
- S1 新手启动期
- S2 稳定执行期
- S3 平台期风险
- S4 风险期
- S5 稳态维持期

### Step 6：判定干预等级

- L0 不干预
- L1 提醒
- L2 调整
- L3 人工

### Step 7：选择动作和触达渠道

动作包括：
- AI 解释
- 任务调整
- 行为难度调整
- 行为替换
- 任务移除
- 饮食方法微调
- 方案切换
- Push / 站内 / 首页 / 短信 / 人工

### Step 8：回写与效果验证

验证窗口建议按动作类型分层：
- 轻提醒：24 小时
- 任务调整： 3 天
- 行为任务调整： 3 天或 1 个周期
- 周复评后微调： 7 天
- 人工干预：由人工 SLA 驱动

## 6. 判定架构

### 6.1 规则引擎直接判定

以下场景由规则引擎直接判定：
- 任务是否逾期
- 连续几天未打卡
- 测评、复评、方案确认是否超时
- Day7/14/21/28 是否到点
- 体重是否缺失到阈值
- 体重是否反弹
- 是否达到阶段目标
- 是否触发 Push
- 是否触发 L3 人工
- 生命周期阈值

### 6.2 AI 语义判定

以下场景由 AI 负责识别：
- 负向情绪
- 信任危机
- 求快心理
- 高危红线表达
- 用户主诉的阻碍类型
- 是否真实执行
- 当前行为任务是否适配
- 外食、夜宵、出差、熬夜等场景

### 6.3 混合判定

以下场景采用混合判定：
- 行为评分
- 平台期归因
- 行为替换建议
- 行为标签状态流转
- 任务难度调整参数
- 是否继续当前行为任务

**混合判定流程**：
1. AI 输出结构化结论
2. 规则引擎读取结构化字段
3. 规则引擎输出最终动作和审计结果

## 7. 统一对象模型

### 7.1 核心状态对象

- 服务方案对象
- 测评对象
- 每日任务对象
- 干预对象
- 订单与履约对象
- 用户生命周期对象

### 7.2 三大分数

| 分数名称 | 字段 | 默认值 | 核心用途 | 驱动内容 |
|----------|------|--------|----------|----------|
| 执行分 | execution_score | 60 | 判断是否在执行 | 任务升级、降级、缩包；驱动 S1/S2/S4 |
| 能量分 | energy_score | 70 | 判断做不做得动 | 驱动运动和行为任务轻重变化；决定 AI 是否切换照护语气 |
| 风险分 | risk_score | 0 | 判断掉队、流失、投诉、安全风险 | 决定 L1/L2/L3；决定是否立即人工介入 |

## 8. 生命周期定义

### 8.1 S1 新手启动期

| 项目 | 内容 |
|------|------|
| 进入条件 | 注册后到方案激活后的前 7 天，或关键数据尚未完整 |
| 重点 | 补档、Day0/Day1 引导、降低摩擦 |

### 8.2 S2 稳定执行期

| 项目 | 内容 |
|------|------|
| 进入条件 | 执行分 >= 65，风险分 < 30，无平台期和高危风险 |
| 重点 | 稳定执行、适度升级，强化反馈 |

### 8.3 S3 平台期风险

| 项目 | 内容 |
|------|------|
| 进入条件 | 真实平台期成立，执行分 >= 60（**平台期判定规则见本节说明**） |
| 重点 | 平台期解释、饮食方法微调、防焦虑和流失 |
| **平台期判定** | 见本节说明（新增详细规则） |

> **平台期判定规则说明**：
> - **真平台期**：连续体重数据≥7天、波动<0.2kg、执行分≥60、能量分≥60、无明显阻碍主诉 → 执行分保护，不扣分
> - **假平台期**：体重缺失≥3天、或执行分<60、或能量分<50、或隐性掉队信号≥2次 → 按正常风险处理

## 12. 干预等级与动作类型

### 12.1 L0 不干预

适用于：风险低、执行稳定、当前无必要额外触达

### 12.2 L1 提醒

适用于：单次未完成、单次数据缺失、轻度拖延、初次复评超时

**动作**：
- 首页卡片
- 站内消息
- AI 轻提醒
- Push

### 12.3 L2 调整

适用于：连续偏差、低能量、平台期、多次提醒无效、行为任务明显不适配

**动作**：
- 缩小任务包
- 任务降级
- 替代任务
- 行为替换
- 饮食方法微调
- 周末模式
- Plan B

### 12.4 L3 人工

适用于：高危红线、医疗边界、投诉退费、连续多次L2无效、高价值用户长期沉默

**动作**：
- 顾问接管
- 工单
- 电话/企微
- 医学或营养师介入

## 13. 全生命周期分阶段PRD

### 13.1 阶段A：首次测评

| 项目 | 内容 |
|------|------|
| 覆盖状态 | NOT_STARTED, IN_PROGRESS, INTERRUPTED, COMPLETED, EXPIRED, CANCELED |
| 监督重点 | 是否进入首次测评、是否中断、是否长时间停留不完成、是否出现非服务适配风险 |
| 显性监督 | 开始测评、中断退出、完成测评、超时未完成、高风险答案命中 |
| 隐性监督 | 多次进入又退出、长时间停留不提交、多次答非所问、连续极短回复 |
| **核心信号** | assessment_resistance(测评抵抗)、assessment_timeout(测评超时)、service_ineligible_risk(不可服务风险) |
| **用户反馈处理** | 任何测评期间负向表达→即时响应+适当引导 |
| 核心动作 | 首次中断→首页恢复卡+AI轻召回；48h内第二次→缩短入口；72h未继续→降低催促；高风险→终止+解释 |

### 13.2 阶段B：报告已出，待继续获取方案

| 项目 | 内容 |
|------|------|
| 监督重点 | 是否理解报告、是否愿意继续获取方案、是否被成本/难度/信任问题阻断 |
| 显性监督 | 报告打开、继续获取方案按钮点击 |
| 隐性监督 | 多次看报告但不继续(>=2次)、高频问"有用吗" |
| **核心信号** | plan_hesitation(方案犹豫)、trust_crisis(信任危机)、urgency_bias(求快心理) |
| **用户反馈处理** | 任何方案相关负向/调整需求→即时AI解释+引导 |
| 核心动作 | 优先解释报告，不先催转化；用AI解答；必要时首页卡承接 |

### 13.3 阶段C：方案前阶段测评P1

| 项目 | 内容 |
|------|------|
| 监督重点 | 是否完成方案前补充信息、是否中断、是否因问题复杂产生疲劳 |
| **用户反馈处理** | 中断后恢复入口；AI追问补齐；多次中断→减少问题量 |
| 核心动作 | 中断后恢复入口；允许AI追问补齐；若多次中断，减少一次性问题量 |

### 13.4 阶段D：方案已出未激活或付费未支付

| 项目 | 内容 |
|------|------|
| 覆盖状态 | READY, AWAITING_PAYMENT |
| 监督重点 | 是否确认方案、是否激活免费方案、是否发起支付但未完成 |
| 显性监督 | 查看方案、方案确认、点击开始执行、点击支付、支付成功或失败 |
| 隐性监督 | 多次看方案不激活、反复问原理但不确认、发起支付后24h不完成 |
| **核心信号** | plan_hesitation(方案犹豫)、payment_abandon(支付放弃) |
| **用户反馈处理** | 支付犹豫→即时解释权益；不想付→问原因 |
| 核心动作 | 免费方案→优先解释后促激活；付费方案→解释权益流程；支付发起未完成→站内卡+Push |

### 13.5 阶段E：付费后待发货、待签收

| 项目 | 内容 |
|------|------|
| 覆盖状态 | AWAITING_DELIVERY |
| 监督重点 | 发货是否正常、物流是否异常、签收是否成功 |
| 核心规则 | 发货和签收以实时回调为主；签收后次日才进入付费Day1 |
| 核心动作 | 发货成功→即时Push；签收成功→即时Push+Day1预告；回调异常→优先对账 |

### 13.6 阶段F：激活后待Day0

| 项目 | 内容 |
|------|------|
| 监督重点 | 用户是否完成Day0准备、是否具备执行条件 |
| 显性监督 | Day0页面打开、Day0任务完成 |
| 隐性监督 | 多次打开Day0页面不执行 |
| **核心信号** | day0_stuck(Day0卡住) |
| **用户反馈处理** | Day0期间任何困难→即时响应+调整 |
| 核心动作 | Day0只强调准备，不做强执行评价；Day0截止前做一次轻提醒；当天未做完，次日首页优先继续承接 |

### 13.7 阶段G：执行期Day1-Day28

| 项目 | 内容 |
|------|------|
| 监督重点 | 今日三件事是否执行、是否出现掉队、低能量、外食、夜宵、失眠、出差等场景 |
| **用户反馈处理** | **全周期最高优先级**：任何负向/调整/求助→即时响应 |
| 核心规则 | 正常→L0或轻反馈；单次未完成→L1；连续未完成→L2；高风险→L3 |

### 13.8 阶段H：Day7/14/21周复评

| 项目 | 内容 |
|------|------|
| 监督重点 | 用户是否点击复评入口、是否完成复评、目标进度是否达标 |
| 核心规则 | 08:00出复评入口；当曰20:00未开始→Push；+24h未开始→再次提醒；+48h未完成→进入超时逻辑 |
| 复评后可调整 | 饮食方法微调、今日任务与下周任务参数微调、行为优先级调整、运动强度轻调 |
| 复评后不可做 | 不切服务阶段、不重生成下一阶段方案 |

### 13.9 阶段I：Day28阶段复评

| 项目 | 内容 |
|------|------|
| 监督重点 | 用户是否进入阶段复评、是否达到当前阶段目标、未达标原因是什么 |
| 核心规则 | Day28 08:00出复评入口；当曰20:00未开始→Push；+24h未开始→Push+首页强提醒；+48h未完成→升级L2 |
| 达标处理 | 当前阶段完成→进入下一服务阶段P2/P3/...→生成下一阶段方案 |
| 未达标处理 | 不直接进入下一阶段→**先归因**→判断是否属于服务内已充分交付→进入服务终局判定→必要时退款 |
| 归因类型 | A1-执行不足、A2-饮食方法不匹配、A3-能量或生理波动、A4-场景干扰过强、A5-产品或履约问题 |

### 13.10 阶段J：终止与退款评估

| 项目 | 内容 |
|------|------|
| 覆盖状态 | TERMINATED |
| 终止原因 | Day28未达标后归因进入终局终止、提前终止规则命中、高风险或投诉导致服务终止 |
| 终止后监督 | 退款流程进度、投诉闭环、人工跟进是否完成、用户是否点击重新开始 |
| 终止后停止 | 不再生成主动执行任务、不再发日常任务型Push、不再推进普通阶段升级 |

| 热量微调 | 体重下降过快/过慢 | ±50-100kcal/日 |
| 餐次调整 | 用户反馈饿/撑 | 3餐→2餐 或 加餐 |
| 食材替换 | 不爱吃/过敏 | 同类食材替换 |
| 做法简化 | 没时间做 | 复杂→简单 |

### 14.2 运动打卡监督

#### 14.2.1 监督指标

| 指标 | 说明 | 阈值 |
|------|------|------|
| 完成率 | 当日运动是否完成 | 完成/未完成 |
| 完成时间 | 是否在计划时间完成 | 计划窗口内 |
| 运动时长 | 实际运动时长 | 目标±20% |
| 强度反馈 | 用户主观感受 | 轻松/刚好/累/极限 |
| 完成质量 | AI评估 | 1-5分 |

#### 14.2.2 异常场景处理

| 异常场景 | 信号 | 处理动作 |
|----------|------|----------|
| 未完成运动 | exercise_missed | L1提醒 |
| 强度过高 | exercise_too_hard | 立即降强度 |
| 强度过低 | exercise_too_easy | 记录，可考虑升级 |
| 受伤/不适 | exercise_injury | 立即暂停+人工关注 |
| 出差/外出 | business_trip_scene | 自动切换轻松模式 |

#### 14.2.3 运动调整动作

| 动作 | 适用场景 | 调整内容 |
|------|----------|----------|
| 降强度 | 反馈累/极限 | 强度降一档 |
| 降频次 | 连续多次未完成 | 3次/周→2次/周 |
| 缩短时长 | 时间不够 | 时长缩短20% |
| 替换运动 | 不喜欢/不适合 | 替换为同类运动 |
| 暂停运动 | 受伤/身体不适 | 暂停3-7天 |

### 14.3 行为打卡监督

#### 14.3.1 监督指标（四个评分维度）

| 维度 | 权重 | 说明 | 分值范围 |
|------|------|------|----------|
| 真实性 | 30% | 是否真实执行 | 1-5分 |
| 执行质量 | 25% | 动作是否规范 | 1-5分 |
| 效果感知 | 20% | 是否感受到效果 | 1-5分 |
| 可持续性 | 25% | 能否长期坚持 | 1-5分 |

**行为评分公式**：
```
behavior_score = 真实性×0.30 + 执行质量×0.25 + 效果感知×0.20 + 可持续性×0.25
```

#### 14.3.2 应用维度分数映射

| behavior_score范围 | 判定 | 建议动作 | 优先级 |
|--------------------|------|----------|--------|
| >=4.2 | 优秀 | 继续巩固/升级难度 | 低 |
| 3.4~4.2 | 良好 | 保持当前 | 中 |
| 2.6~3.4 | 一般 | 难度调整 | 中 |
| 1.8~2.6 | 较差 | 行为替换 | 高 |
| <1.8 | 极差 | 任务移除 | 最高 |

#### 14.3.3 周末任务与打卡调整联动

| 场景 | 处理规则 |
|------|----------|
| 周末行为任务打卡 | 仅保留已完成1轮的习惯性任务 |
| 周末打卡未完成 | 不触发"连续未完成"惩罚 |
| 周末期间评分 | 不受周末模式影响，正常评分 |
| 用户周末反馈调整 | 按正常流程处理 |

## 15. 行为干预任务PRD

### 15.1 任务定位

行为干预是"今日三件事"中的第3个任务。任务来源于行为库，分两类：

| 任务类型 | 执行周期 | 目标 | 结束条件 |
|----------|----------|------|----------|
| 数据追踪类 | 1天 | 获取数据 | 完成后结束 |
| 行为习惯类 | 3天 | 形成习惯 | Day3后输出行为评分和动作 |

### 15.2 行为追问的四个评分维度

| 维度 | 权重 | 说明 |
|------|------|------|
| 真实性 | 30% | 用户是否真实执行 |
| 执行质量 | 25% | 执行动作是否规范 |
| 效果感知 | 20% | 用户是否感受到效果 |
| 可持续性 | 25% | 用户是否认为可以长期坚持 |

**建议公式**：
```
behavior_score = 0.30*真实性 + 0.25*执行质量 + 0.20*效果感知 + 0.25*可持续性
```

### 15.3 Day1/Day2/Day3目标

| 阶段 | 目标 |
|------|------|
| Day1 | 确认是否真实执行；判断是否明显不适配 |
| Day2 | 判断执行质量；判断是否开始产生效果；识别具体阻碍 |
| Day3 | 判断是否可形成习惯；形成最终行为评分 |

### 15.4 3天评估动作矩阵

| 完成情况 | 判定 | 动作 |
|----------|------|------|
| 0/3或1/3完成，且无回复 | 隐性掉队 | 当前任务退出执行位→标签保留→当前任务降入P3→次日切换行为库下一既有任务 |
| 0/3或1/3完成，但用户明确说"很难/没时间/不想做" | 核心动作仍合理 | 先难度调整 |
| | 核心动作不合理但目标合理 | 行为替换 |
| | 用户明确拒绝该方向 | 任务移除 |
| 2/3完成，但Day3失败 | - | 优先难度调整；多次同类阻碍再做替换 |
| 3天完成，behavior_score>=4.2 | 优秀 | 继续巩固或视为完成→流转下一任务→对应负向标签进入weakened |
| 3天完成，3.4<=behavior_score<4.2 | 一般 | 继续当前任务→不升级难度 |
| 3天完成，2.6<=behavior_score<3.4 | 较差 | 优先难度调整 |
| 3天完成，1.8<=behavior_score<2.6 | 差 | 优先行为替换→新任务进入P1顶部 |
| 3天完成，behavior_score<1.8 | 极差 | 当前任务移除→次日切换下一既有任务→标签保留 |

### 15.5 调整动作类型详解

#### 15.5.1 难度调整规则

**定义**：在不改变目标和核心动作的前提下调参数。

**可调整参数**：
- 次数（frequency）
- 频次（interval）
- 时长（duration）
- 执行窗口（time_window）
- 执行场景（context）

**示例**：
| 原任务 | 调整后 |
|--------|--------|
| 饭后站立10分钟 | 饭后站立3分钟 |
| 每天做3次深呼吸 | 每天做1次 |
| 睡前1小时关手机 | 先提前15分钟 |
| 每天喝8杯水 | 每天喝4杯水 |

**触发条件**：
- behavior_score 在 2.6 ~ 3.4 之间
- 用户反馈"有点难"
- timing_friction 信号触发

#### 15.5.2 行为替换规则

**定义**：行为替换**不是**从行为库直接选一个旧任务，而是 AI 基于原任务目标生成一个更好执行的新任务。

**AI生成新任务的逻辑流程**：

```
Step 1: 分析原任务
  - 提取原任务的核心健康目标
  - 分析用户反馈的阻碍原因
  - 评估当前场景适配度

Step 2: 生成替换方案
  - 约束1: 目标不变（保持核心健康目标）
  - 约束2: 成本更低（降低执行门槛）
  - 约束3: 更适配当前场景（匹配用户生活状态）

Step 3: 生成新任务
  - AI根据约束条件生成新任务
  - 新任务进入P1顶部（最高优先级）
  - 记录替换原因
```

**AI行为替换示例**：

| 原任务 | 阻碍原因分析 | 替换后新任务 | 目标保持 |
|--------|--------------|--------------|----------|
| 每天100个开合跳 | 用户反馈膝盖疼 | 每天10分钟散步 | 有氧运动 |
| 睡前冥想15分钟 | 用户反馈没时间 | 午间深呼吸3次 | 放松减压 |
| 记录每餐热量 | 用户反馈太复杂 | 只记录午餐热量 | 热量 awareness |
| 每天做瑜伽30分钟 | 用户反馈坚持不了 | 每天拉伸5分钟 | 身体活动 |
| 睡前不看手机1小时 | 用户反馈睡不着时想看 | 改为醒来后不看手机 | 减少蓝光接触 |

**替换后处理**：
- 新任务进入 **P1 顶部**（最高优先级）
- 原任务标记为 `replaced`
- 记录替换原因

**触发条件**：

| 条件 | 分值范围 | 动作 |
|------|----------|------|
| 行为评分较低 | 1.8 <= behavior_score < 2.6 | 优先行为替换 |
| 核心动作不合理但目标合理 | - | 行为替换 |
| 用户明确拒绝该方向 | - | 任务移除 |

#### 15.5.3 任务移除规则

**以下情况可移除**：
- Day1 明确反馈"太简单/没必要/不适合"
- 连续不打卡且不回应（>=3天）
- 评分极低（behavior_score < 1.8）
- 用户明确拒绝继续

**移除前必须执行**：
1. 必须解释原因（AI话术）
2. 必须让用户确认（需要用户交互确认）

**移除后处理**：
- 次日切换下一既有任务
- 当前任务记为 `removed`
- 标签保留在 confirmed 或 intervening

## 16. 方案调整规则（新增）

### 16.1 方案调整触发条件

| 触发类型 | 触发场景 | 处理流程 |

| 调整类型 | 适用范围 | 调整幅度 |

| 方案切换 | 主方案更换 | 需复评后决策 |

### 16.3 方案调整流程

```
用户/系统触发方案调整
    ↓

```

## 17. 用户主动反馈收集与处理（新增）

> **核心原则**：用户在任何阶段、任何流程、任何闲聊时主动表达的负向情绪、方案调整需求、任务调整需求，都是最高优先级的监督输入，必须立即响应。

### 17.1 用户主动反馈类型

| 反馈类型 | 信号Code | 触发场景 | 响应优先级 |
|----------|----------|----------|------------|
| 负向情绪表达 | user_negative_emotion | 用户说"太难了"、"想放弃"、"心情不好"、"压力大"等 | **P0-即时** |
| 方案调整请求 | user_plan_adjust_request | 用户说"想换方案"、"目标完不成"、"想调整计划"等 | **P0-即时** |
| 任务调整请求 | user_task_adjust_request | 用户说"任务太多了"、"这个做不来"、"想换个任务"等 | **P1-<1h** |
| 用户求助 | user_help_request | 用户说"怎么办"、"帮帮我"、"有问题"等 | **P0-即时** |
| 正向反馈 | user_positive_feedback | 用户说"很有用"、"效果很好"等 | 记录存档 |

### 17.2 用户主动反馈处理流程

```
用户主动输入（任何时刻）
    ↓
AI实时语义识别（判断是否为负向/调整/求助）
    ↓
是 → 生成对应信号 → 立即触发干预（不等待定时任务）
    ↓
否 → 判断为闲聊/正向反馈 → 正常响应或存档
```

### 17.3 不同反馈类型的处理策略

#### 17.3.1 负向情绪处理

| 场景 | 处理策略 |
|------|----------|
| 轻度负向（偶尔抱怨） | AI共情+鼓励，次日轻关注 |
| 中度负向（持续沮丧） | AI共情+任务减负（L2），48h内持续关注 |
| 重度负向（想放弃/自暴自弃） | AI共情+立即缩小任务包+人工关注准备 |
| 高危表达（涉及安全） | 不等待，直接升级L3人工 |

#### 17.3.2 方案调整请求处理

| 场景 | 处理策略 |
|------|----------|
| 用户觉得目标太高 | AI解释当前进度合理性，若坚持，收集归因后进入方案调整流程 |
| 用户觉得方案不适合 | AI追问具体原因，判断是执行问题还是方案问题 |
| 用户想换方案 | 记录原因，评估是否满足换方案条件，进入方案调整流程 |

#### 17.3.3 任务调整请求处理

| 场景 | 处理策略 |
|------|----------|
| 任务太多做不完 | 立即缩小任务包（从3件减到2件） |
| 任务太难 | 立即难度调整或行为替换 |
| 任务不想做 | 追问原因，根据原因选择调整或替换 |
| 时间安排不合理 | 调整提醒时间和任务窗口 |

#### 17.3.4 用户求助处理

| 场景 | 处理策略 |
|------|----------|
| 一般问题 | AI即时回答，记录为知识库候选 |
| 复杂问题 | AI回答后标记，48h后回访确认 |
| 医疗/专业问题 | 立即升级L3人工 |

### 17.4 用户反馈的分数影响

| 反馈类型 | 对执行分 | 对能量分 | 对风险分 |
|----------|----------|----------|----------|
| 负向情绪 | - | -5~-15 | +5~+20 |
| 方案调整请求 | 记录 | - | +10 |
| 任务调整请求 | 记录 | - | - |
| 用户求助 | - | - | - |

### 17.5 全阶段用户反馈覆盖

| 阶段 | 用户反馈场景 | 响应策略 |
|------|--------------|----------|
| 测评期 | 测评太难/不想测了 | 即时引导+简化测评 |
| 方案期 | 方案太贵/不适合 | 即时解释+方案调整评估 |

## 19. 标签流转规则

### 19.1 标签状态

| 状态 | 说明 |
|------|------|
| suspected | 怀疑/待确认 |
| confirmed | 已确认 |
| intervening | 干预中 |
| weakened | 弱化中 |
| resolved | 已解决 |

### 19.2 标签进入

| 进入条件 | 触发 |
|----------|------|
| 测评命中 | 测评直接命中 |
| 主动问询承认 | 用户主动承认 |
| AI语义识别明确主诉 | AI明确识别 |
| 隐性信号重复出现>=2 | 系统检测 |

### 19.3 标签升级

| 触发条件 |
|----------|
| 7天内同类问题重复出现 |
| 当前行为任务失败 |
| 隐性和显性反馈同时命中 |

### 19.4 标签弱化

满足任一即可：
- 连续1个行为周期 behavior_score >= 4.2 且 7天内无重复负向信号

### 19.5 标签解决

满足以下组合：
- 连续2个相关任务周期表现良好
- 或连续14天无相关显性和隐性信号
- 若有客观数据，不恶化

### 19.6 标签保留

以下情况下标签保留在 confirmed 或 intervening：
- 任务未完成
- 行为评分低
- 用户明确拒绝
- 隐性负向信号仍持续

## 22. 验收标准

### 22.1 监督有效

- 系统能识别显性与隐性监督事件
- 用户主动反馈能被即时响应
- 体重、签收等关键事件实时优先
- 定时检测只做缺失和兜底

### 22.2 判定可解释

- 能说清楚某次动作为什么触发
- 能回放 event → signal → score → action

### 22.3 动作可执行

- 任务调整能落到实际任务
- 行为替换能生成新任务并进入P1
- 终止和退款评估有明确入口

### 22.4 渠道不打扰

- Push有冷却和优先级
- 高风险优先，普通提醒克制

### 22.5 全生命周期打通

- 首次测评、方案、执行、复评、阶段流转、终止全部被覆盖
