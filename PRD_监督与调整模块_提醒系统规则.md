## 一、产品概述

### 1.1 产品定位

监督与调整模块的提醒系统旨在通过多渠道（APP推送、短信、人工召回）触达用户，确保用户按时完成健康行为打卡、回答AI问询、查看方案等任务，同时遵循"低打扰原则"，避免过度打扰用户。

### 1.2 核心目标

1. **数据采集完整性**：通过主动问询补齐用户健康数据
2. **行为执行监督**：提醒用户完成每日健康打卡任务
3. **用户召回**：对长时间未活跃用户进行召回
4. **干预调整**：根据用户反馈动态调整健康方案

## 三、触发规则

### 3.1 打卡任务提醒触发

#### 3.1.1 日常打卡提醒

| 触发条件 | 提醒渠道 | 提醒时间 |
|----------|----------|----------|
| 每日任务未在18:00前完成 | PUSH | 当日18:00 |
| 连续2天未打卡 | PUSH | 次日10:00 |
| 连续3天未打卡 | 短信 | 第三日10:00 |
| 连续7天未打卡 | 人工召回 | 第七日 |

#### 3.1.2 打卡完成后的追问触发

根据《行为打卡后追问规则》和《运动打卡反馈收集》：

| 触发条件 | 动作 | 追问轮次 |
|----------|------|----------|
| 用户完成打卡 | AI给予正向反馈 + 追问 | 最多3轮 |
| 用户未回复 | 不再追问 | - |
| 用户回复1次 | 最多追问1轮 | 1轮 |
| 用户持续互动 | 最多3轮 | 3轮 |

### 3.2 主动问询触发

根据《日常主动询问规则》：

#### 3.2.1 触发前置条件

```
前置条件 = (用户当前无更高优先级事项) 
         AND (pending_questions 非空) 
         AND (距上次日常问询 > 3小时)
```

#### 3.2.2 具体触发场景

| 场景 | 触发条件 | 提醒渠道 |
|------|----------|----------|
| 用户打开APP时 | 待回答问题池非空 | 实时推送 |
| 初始问卷缺失字段 | 有未补齐的必填项 | 实时推送 |
| 行为习惯库待筛查 | 高风险行为未筛查 | 实时推送 |
| 用户主动聊天 | AI可自然插入问询 | 对话中 |

### 3.3 方案查看提醒

根据《查看方案》：

| 触发条件 | 提醒渠道 | 提醒时间 |
|----------|----------|----------|
| 系统复评前3天 | PUSH | 复评前3天12:00 |
| 复评当天 | PUSH | 当日10:00 |
| 新方案生成 | PUSH | 实时推送 |

## 四、提醒渠道规则

### 4.1 APP推送(PUSH)规则

#### 4.1.1 推送优先级

| 优先级 | 场景 | 推送时间 | 推送内容 |
|--------|------|----------|----------|
| P1 | 初始问卷缺失字段补齐 | 实时 | "您的健康档案还需完善以下信息..." |
| P2 | 日常问询问题 | 实时 | "想了解一下您的情况..." |
| P3 | 行为习惯筛查 | 实时 | "帮您评估一下健康习惯..." |
| P4 | 打卡任务提醒 | 定时18:00 | "今日健康任务待完成..." |
| P5 | 方案复评提醒 | 定时10:00 | "距复评还有X天..." |

#### 4.1.2 推送限制

- **每日推送上限**：不超过5条
- **每小时推送上限**：不超过2条
- **免打扰时段**：22:00 - 次日8:00（除非紧急）

#### 4.1.3 推送内容规范

```
标题：[健康助手]
内容结构：共情 + 动作 + 鼓励
示例：
- "看到您今天还没完成打卡，慢慢来～点击这里完成今日任务吧💪"
- "您的饮食记录需要补充一点点信息，点这里完善一下吧～"
```

### 4.2 短信提醒规则

#### 4.2.1 短信触发场景

| 触发条件 | 短信内容 | 发送时间 |
|----------|----------|----------|
| 连续3天未打卡 | "您已3天未完成健康任务，点击链接查看：" | 10:00 |
| 初始问卷未完成 | "您的健康档案尚未完善，点击链接补充：" | 10:00 |
| 方案复评提醒 | "您的健康方案即将复评，点击查看：" | 10:00 |

#### 4.2.2 短信频率限制

- **每周最多发送**：2条
- **同类型短信**：7天内不重复发送
- **用户可退订**：回复"T"退订

#### 4.2.3 短信内容模板

```
【健康助手】尊敬的 用户，您已连续3天未完成健康打卡。我们关心您的健康，点击链接查看今日任务：XXX
```

### 4.3 人工召回规则

#### 4.3.1 召回触发条件

| 触发条件 | 召回方式 | 处理时限 |
|----------|----------|----------|
| 连续7天未活跃 | AI外呼 | 7天内 |
| 连续14天未活跃 | 人工外呼 | 14天内 |
| 连续30天未活跃 | 转交人工客服 | 立即 |

#### 4.3.2 召回流程

```
用户失联判断 → 触发召回 → AI外呼(3次) → 人工外呼(2次) → 转交客服
     │            │           │            │            │
     ▼            ▼           ▼            ▼            ▼
  7天未活跃   10:00-20:00  每次间隔2h   每次间隔4h    及时处理
```

#### 4.3.3 召回话术要求

1. **友好开场**："您好，我是健康助手的人工客服..."
2. **说明来意**："注意到您一段时间没使用，我们很关心..."
3. **了解原因**："方便问一下是什么原因吗？"
4. **提供帮助**："有什么我们可以帮您的？"
5. **记录反馈**："感谢您的反馈，我们会改进..."

## 五、频率与时机规则

### 5.1 每日问询限制

根据《日常主动询问规则》：

| 规则项 | 限制值 |
|--------|--------|
| 每日最多主动问询次数 | ≤3次 |
| 单次问询最多问题数 | 1个问题 |
| 每次问询后最多追问轮次 | 1轮 |
| 单次对话总轮次 | ≤3轮 |
| 两次问询最小间隔 | ≥3小时 |

### 5.2 打卡追问限制

根据《行为打卡后追问规则》：

| 场景 | 最大追问轮次 |
|------|--------------|
| 用户未回复 | 不再追问 |
| 用户回复1次 | 最多1轮 |
| 用户持续互动 | 最多3轮 |
| 用户主动提问 | 不限制 |

### 5.3 冷却机制

根据《日常主动询问规则》：

| 情况 | 冷却时间 |
|------|----------|
| 用户超时未回复（5分钟） | 3天后可再次询问 |
| 用户未提供答案但未拒绝 | 7天后再次询问 |
| 用户明确拒绝 | 14天内不再询问 |
| 周期性问题已回答 | 7天后再次询问 |

## 六、优先级策略

### 6.1 提醒优先级

| 优先级 | 场景 | 处理规则 |
|--------|------|----------|
| **P1** | 初始问卷缺失字段 | 立即处理 |
| **P2** | 日常与客户沟通问题 | 排队处理 |
| **P3** | 行为习惯筛查 | 按健康获益得分排序 |
| **P4** | 打卡任务提醒 | 定时处理 |
| **P5** | 方案复评提醒 | 提前3天提醒 |

### 6.2 多任务排队规则

当存在多个待处理任务时：

```
优先级排序规则：
1. P1 > P2 > P3 > P4 > P5
2. 同优先级按健康获益得分排序
3. 同优先级同得分按执行难度排序（难的优先）
```

## 七、用户状态识别与响应

### 7.1 问询状态识别

根据《日常主动询问规则》：

| 用户状态 | 定义 | 系统动作 | 后续策略 |
|----------|------|----------|----------|
| **answered** | 提供有效回答 | 结束问询，数据入库 | 一次性问题不再询问；周期性冷却7天 |
| **skip** | 未提供答案但未拒绝 | 结束问询，不入库 | 冷却7天后再次询问 |
| **refuse** | 明确拒绝回答 | 停止问询，不入库 | 冷却14天内不再询问 |
| **timeout** | 超5分钟未回复 | 结束问询，不入库 | 3天后可再次询问 |
| **unrecognized** | 回复无法识别 | AI澄清一次 | 再识别一次仍失败则返回数据池 |
| **topic_shift** | 转移话题 | 顺着新话题继续 | 问题返回数据池，冷却7天 |

### 7.2 打卡反馈识别

根据《运动打卡反馈收集》：

| 用户反馈 | 系统响应 |
|----------|----------|
| 轻松 | 鼓励，结束反馈收集 |
| 刚好 | 鼓励，结束反馈收集 |
| 挺累 | 追问具体项目，评估是否调整方案 |
| 极限了 | 追问具体项目，建议降低强度 |

## 八、异常处理

### 8.1 提醒发送异常

| 异常情况 | 处理方式 |
|----------|----------|
| PUSH发送失败 | 降级为短信 |
| 短信发送失败 | 记录日志，转入人工处理 |
| 用户手机号无效 | 标记，转入人工召回 |
| 超过发送频率限制 | 加入队列，等待发送 |

### 8.2 用户响应异常

| 异常情况 | 处理方式 |
|----------|----------|
| 用户超5分钟未回复 | 结束当前问询，3天后再次尝试 |
| 用户连续2次无效回复 | 停止追问，根据现有信息估算 |
| 用户表现不耐烦 | 停止追问，记录偏差风险 |
| 用户明确拒绝 | 停止当前任务，标记用户偏好 |

### 8.3 数据同步异常

| 异常情况 | 处理方式 |
|----------|----------|
| 数据入库失败 | 重试3次，失败后标记待处理 |
| 状态更新失败 | 回滚操作，记录日志 |

### 9.1 需要记录的数据

### 9.2 数据记录规则

> 以下任务**必须在首轮追问具体数据**：

## 十、方案调整联动

### 10.1 触发调整条件

根据《行为打卡后追问规则》：

| 条件 | 调整动作 |
|------|----------|
| 连续3天打卡完成且评分≥3 | 移除负向行为标签 |
| 未完成3天打卡且评分＜3 | 任务流转到P3列表二次巩固 |
| 用户反馈任务困难 | 询问是否更换任务 |
| 用户持续负向情绪 | 触发人工干预 |

### 10.2 调整流程

```
用户反馈 → AI识别 → 评估困难程度 → 给出替代方案/调整强度 → 用户确认 → 更新方案
    │         │            │              │              │           │
    ▼         ▼            ▼              ▼              ▼           ▼
 打卡完成  评分计算    P1:情绪疏导    P2:降低难度    P3:更换任务   数据同步
```

## 十一、禁止行为

根据各源文档的约束：

1. **禁止强制推送**：不得在用户明确拒绝后继续推送
2. **禁止医学诊断**：AI不得提供医学诊断、治疗方案或用药建议
3. **禁止连续追问**：单次对话追问不超过3轮
4. **禁止夜间打扰**：22:00-次日8:00不发送非紧急推送
5. **禁止机械表达**：问询话术需自然，避免问卷感
